# 01. 왜 프론트와 백엔드를 나눌까

> **이 문서에서 배우는 것**
> - 지금까지 만든 방식(서버 렌더링)과 분리 구조(SPA + API)의 차이
> - 분리했을 때 얻는 것과 **잃는 것** (분리가 항상 정답은 아닙니다!)
> - 회사 실무(eGovFrame)와 최신 트렌드가 어떻게 이어지는지

---

## 1. 지금까지의 방식 — 서버가 화면까지 만들어 준다

intro-jpa에서 목록 화면이 뜨는 과정을 다시 그려 보면:

```
[브라우저]                         [스프링부트 서버 (8080)]
    │                                      │
    │  ── GET / ──────────────────────▶   │ ① 컨트롤러가 요청 접수
    │                                      │ ② 서비스 → DB에서 목록 조회
    │                                      │ ③ 타임리프가 list.html에 데이터를 끼워
    │                                      │    "완성된 HTML"을 조립
    │  ◀────────── 완성된 HTML ──────      │
    │  ④ 받은 HTML을 그대로 표시            │
```

화면(HTML)이 **서버에서** 완성됩니다. 그래서 이 방식을
**서버 사이드 렌더링(SSR, Server-Side Rendering)** 이라고 부릅니다.
링크를 누를 때마다 새 HTML을 통째로 다시 받아오고, 그때마다 화면이 깜빡입니다.

> 🏢 **eGovFrame 연결 포인트** — 회사 실무의 JSP도 정확히 이 방식입니다.
> 타임리프 자리에 JSP가 있을 뿐, "서버가 데이터를 끼워 완성된 HTML을 준다"는
> 그림은 완전히 같습니다. 지금까지 배운 것이 곧 실무의 기본 구조입니다.

---

## 2. 분리 구조 — 서버는 데이터만, 화면은 브라우저가

이번 모듈에서 만들 구조입니다:

```
[브라우저 (React 앱)]               [스프링부트 API 서버 (8080)]
    │                                      │
    │  처음 한 번: React 앱(JS 뭉치)을 받아옴 │
    │     (개발 중엔 5173 개발 서버가 줌)     │
    │                                      │
    │  ── GET /api/intros ────────────▶    │ ① 컨트롤러가 요청 접수
    │                                      │ ② 서비스 → DB에서 목록 조회
    │  ◀── 데이터만(JSON) ─────────────     │ ③ JSON으로 변환해서 응답 (HTML 없음!)
    │                                      │
    │  ④ React가 JSON을 받아                │
    │     브라우저 안에서 화면을 조립          │
```

서버는 이제 화면을 모릅니다. `[{"id":1,"name":"김인턴",...}]` 같은
**날것의 데이터(JSON)** 만 응답하고, 그걸 사람이 볼 수 있는 화면으로 바꾸는 일은
브라우저에서 도는 JavaScript(React)가 합니다. 화면 조립이 클라이언트(브라우저)에서
일어나므로 **클라이언트 사이드 렌더링(CSR)** 이라 부릅니다.

이때 React 앱은 HTML 파일을 **딱 하나만** 갖습니다(빈 `<div>` 하나뿐인 index.html).
화면 전환은 페이지 이동이 아니라 JS가 그 div 안의 내용을 갈아끼우는 것입니다.
그래서 이런 앱을 **SPA(Single Page Application, 단일 페이지 앱)** 라고 합니다.

---

## 3. 왜 나눌까 — 중요한 이유부터

### ① 클라이언트가 하나가 아니기 때문 (가장 큰 실무적 이유)

서비스가 커지면 웹만 있는 게 아닙니다. 안드로이드 앱, 아이폰 앱, 관리자 화면,
다른 회사 시스템까지 같은 데이터를 씁니다.

```
              ┌─ 웹 (React)
[API 서버] ──┼─ 안드로이드 앱
 (하나!)      ├─ 아이폰 앱
              └─ 제휴사 시스템
```

서버가 HTML을 만들어 버리면 웹 브라우저만 쓸 수 있지만, **JSON을 주면 누구나 쓸 수
있습니다.** 백엔드 하나로 모든 클라이언트를 감당하는 것 — 이게 API 분리의 핵심 가치입니다.

### ② 역할 분담이 명확해진다

화면 전문(프론트엔드 개발자)과 데이터·업무 로직 전문(백엔드 개발자)이
**서로의 코드를 건드리지 않고** 동시에 일할 수 있습니다. 둘 사이의 약속이 바로
API 명세("`GET /api/intros`를 부르면 이런 JSON을 준다")이고, 이 약속만 지키면
한쪽을 통째로 갈아엎어도 다른 쪽은 영향이 없습니다.

실제로 우리 샘플에서 intro-jpa → intro-api로 바꿀 때 **서비스와 리포지토리 코드는
한 글자도 바뀌지 않았습니다.** 화면 기술이 Thymeleaf에서 React로 완전히 교체됐는데도요.

### ③ 사용자 경험 — 필요한 부분만 다시 그린다

SSR은 글 하나를 지워도 페이지 전체를 새로 받습니다. SPA는 목록에서 그 줄만
사라지게 할 수 있습니다. 페이스북, 유튜브, 지메일처럼 "앱같이 부드러운" 웹은
전부 이 방식입니다.

### ④ 독립적인 배포와 확장

프론트만 고쳤으면 프론트만 배포하면 됩니다. 접속자가 몰리면 API 서버만 늘리면 됩니다.

## 4. 그럼 무조건 나누는 게 좋은가? — 아닙니다

분리에는 **비용**이 따릅니다. 정직하게 짚고 갑니다.

| 잃는 것 | 설명 |
|---|---|
| **복잡도 증가** | 프로젝트 2개, 서버 2개, 빌드 도구 2개(Gradle + npm). 배우고 관리할 게 두 배 |
| **CORS 등 새로운 문제** | 출처(포트)가 달라지는 순간 브라우저 보안 정책과 싸워야 합니다 (04 문서에서 직접 겪습니다) |
| **첫 화면이 느릴 수 있음** | JS 뭉치를 다 받아야 화면이 나옵니다. SSR은 첫 HTML이 바로 보입니다 |
| **검색 노출(SEO) 불리** | 검색 로봇이 "빈 div"만 볼 수 있습니다 (해결 기술은 있지만 또 복잡해집니다) |

그래서 실무 판단 기준은 대략 이렇습니다:

- **사내 관리자 화면, 단순 게시판, 콘텐츠 위주 사이트** → 서버 렌더링으로 충분한 경우가 많음
- **여러 클라이언트가 필요하거나, 화면 상호작용이 많은 서비스** → 분리 구조가 유리

"신기술이니까 무조건 분리"가 아니라 **상황에 맞는 구조를 고르는 것**이 실력입니다.
회사의 eGovFrame(SSR)도 낡은 게 아니라, 행정 업무 시스템이라는 상황에 맞는 선택인 것입니다.

> 📌 참고로 두 방식은 섞어 쓸 수도 있습니다. JSP 페이지 안에서 일부 영역만
> Ajax로 JSON을 받아 갱신하는 패턴은 eGovFrame 실무에서도 아주 흔합니다.
> 이번 모듈을 마치면 그 코드가 "작은 SPA"로 보일 겁니다.

## 5. 우리가 만들 것

intro-jpa 하나를 이렇게 둘로 나눕니다:

```
intro-jpa (서버 렌더링)          →        intro-api (백엔드)        intro-react (프론트엔드)
─────────────────────                    ──────────────────        ────────────────────
Controller (화면 이름 반환)       →        RestController (JSON 반환)
Service                          →        Service (그대로!)
Repository                       →        Repository (그대로!)
templates/*.html (타임리프)       →        (삭제)                     src/components/*.jsx
static/css                       →        (삭제)                     src/index.css
```

| 계층 | 어떻게 되나 |
|---|---|
| 컨트롤러 | HTML 대신 JSON을 반환하도록 **수정** → [02 문서](./02_REST_API_기본개념.md) |
| 서비스/리포지토리/엔티티 | **그대로** — 계층을 나눠 둔 보람! |
| 화면(템플릿) | React 컴포넌트로 **재작성** → [03](./03_React_시작하기.md), [04 문서](./04_React로_CRUD_화면_만들기.md) |

다음 문서에서 두 진영의 "공용어"인 REST API부터 배웁니다.
