# 03. 에러 처리와 실무 고려사항 — "안 될 때"를 다루기

> **이 문서에서 배우는 것**
> - 외부 호출이 실패하는 **세 가지 종류**와 각각의 대처
> - RestClient의 상태 코드 처리(`onStatus`), 타임아웃, 간단한 재시도
> - **API 키를 코드에 박지 않고 안전하게 다루기** (실무 필수, 사고 최다 발생 지점)
> - 회사에서 쓰는 **공공데이터포털(data.go.kr)** 연동으로 확장하기

**결론부터(중요도 순):**
> 1. **키 보안이 1순위다.** 키를 소스에 박고 깃허브에 올리는 사고가 신입에게 가장 흔하고, 피해가 가장 큽니다.
> 2. **외부는 반드시 실패한다고 가정하라.** 타임아웃·예외 처리 없는 외부 호출은 "장애를 부르는 코드"다.
> 3. **실패를 종류별로 구분하라.** 내 입력 잘못 / 상대 서버 잘못 / 일시적 네트워크 문제는 대처가 다르다.

---

## 1. 외부 호출은 반드시 실패한다 — "잘 될 때"만 짜면 안 되는 이유

우리 코드가 완벽해도 외부 API는 우리 통제 밖입니다. 언제든 이런 일이 생깁니다.

- 사용자가 없는 도시("서울시청 앞")를 입력함
- 외부 서버가 잠깐 점검 중(503)이거나 과부하(500)
- 네트워크가 느려 5초가 지나도 응답이 없음(타임아웃)
- 무료 API의 하루 호출 한도를 초과함(429)

이걸 처리 안 하면 사용자는 **의미 없는 500 에러 화면**을 보고, 심하면 우리 서버가 멈춥니다.

---

## 2. 실패의 세 종류와 대처 — 우리 샘플의 처리

컨트롤러에서 예외를 **종류별로 나눠** 잡는 이유가 여기 있습니다.

| 종류 | 원인 | 누구 잘못 | 대처 | 우리 코드 |
|---|---|---|---|---|
| ① 입력이 잘못됨 | 없는 도시, 오타 | **사용자** | 고쳐 달라고 **안내** | `CityNotFoundException` → 노란 안내 박스 |
| ② 상대 서버 오류 | 외부 API 4xx/5xx | **외부(또는 우리 요청)** | "잠시 후 재시도" 안내, 로그 남김 | `RestClientException` → 안내 박스 |
| ③ 네트워크/지연 | 타임아웃, 연결 끊김 | **아무도(일시적)** | 타임아웃 + (선택)재시도 | `SimpleClientHttpRequestFactory` 타임아웃 |

```java
try {
    model.addAttribute("weather", weatherService.getWeather(city));
} catch (CityNotFoundException e) {          // ① 사용자 입력 문제
    model.addAttribute("error", e.getMessage());
} catch (RestClientException e) {            // ②③ 외부/네트워크 문제 (타임아웃도 이걸로 잡힘)
    model.addAttribute("error", "날씨 서버에 연결하지 못했습니다. 잠시 후 다시 시도해 주세요.");
}
```

> ⚠️ **함정 — "존재하지 않는 데이터"를 500으로 흘려보내지 마세요.** 없는 도시를 그냥 두면
> 서버 예외(500)로 나갑니다. 하지만 "서버가 고장난 것"과 "찾는 값이 없는 것"은 **다른 상황**입니다.
> 사용자에게는 "고쳐서 다시 입력하세요"라고 안내해야지, "서버 오류"라고 하면 안 됩니다.

> 📌 **RestClient의 기본 동작** — 4xx/5xx 응답이 오면 RestClient는 **예외를 던집니다**
> (`RestClientResponseException`, 위 `RestClientException`의 하위). 정상(2xx)일 때만 `body(...)`가
> 값을 줍니다. 즉 "성공했겠지" 하고 안심해도, 실패는 예외로 알아서 튀어나옵니다.

### 상태 코드별로 세밀하게: `onStatus`

"404는 이렇게, 500은 저렇게" 나누고 싶으면 `onStatus`로 가로챌 수 있습니다.

```java
restClient.get().uri("/v1/forecast?...")
    .retrieve()
    .onStatus(status -> status.value() == 429,          // 호출 한도 초과라면
        (req, res) -> { throw new TooManyRequestsException("잠시 후 다시 시도하세요"); })
    .body(ForecastResponse.class);
```

## 3. 타임아웃 — 없으면 장애가 전염된다

우리 샘플은 [RestClientConfig](./샘플/weather/src/main/java/com/example/weather/config/RestClientConfig.java)에서
연결 3초, 읽기 5초 타임아웃을 걸어 뒀습니다.

```java
SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
factory.setConnectTimeout(Duration.ofSeconds(3));   // 연결까지 최대 3초
factory.setReadTimeout(Duration.ofSeconds(5));      // 응답까지 최대 5초
```

**왜 필수인가:** 타임아웃이 없으면, 외부 서버가 응답을 안 줄 때 그 요청을 처리하던
**우리 서버 스레드가 영원히 붙잡힙니다.** 그런 요청이 몇십 개 쌓이면 스레드가 고갈되어
**멀쩡하던 다른 기능까지 다 멈춥니다.** 외부의 장애가 우리에게로 **전염**되는 것이죠.
타임아웃은 "적당히 포기하고 스레드를 돌려받는" 안전장치입니다.

> 📌 **재시도(retry)는 신중하게.** "실패하면 한 번 더" 가 늘 옳지는 않습니다.
> - ✅ 재시도해도 되는 것: **조회(GET)**, 일시적 네트워크 오류/503
> - ❌ 재시도하면 위험한 것: **결제·주문 같은 쓰기(POST)** — 두 번 결제될 수 있음!
> 재시도가 정말 필요하면 스프링의 `spring-retry`(`@Retryable`)를 쓰되, 위 구분을 반드시 지키세요.

## 4. API 키를 안전하게 다루기 — 🔴 가장 중요

우리가 쓴 Open-Meteo는 키가 필요 없었지만, **실무 API는 대부분 키가 필요합니다.**
그리고 이 키를 잘못 다뤄서 나는 사고가 신입에게 **가장 흔하고 피해가 큽니다.**

### 절대 하면 안 되는 것

```java
// 🚫 절대 금지: 키를 소스 코드에 박기
String serviceKey = "a1b2c3d4-1234-5678-abcd-ef0123456789";  // ← 이러면 안 됨!
```

**왜 위험한가:**
- 이 파일을 깃허브에 올리는 순간, 키가 **전 세계에 공개**됩니다.
- 공개 저장소의 API 키는 **봇이 몇 분 안에 자동으로 긁어갑니다.** 남이 내 키로 API를 마구 써서
  요금 폭탄을 맞거나, 유료 한도가 소진되거나, 키가 차단됩니다.
- **한 번 커밋하면 지워도 소용없습니다.** git 히스토리에 영원히 남기 때문에, 키를 **폐기·재발급**해야 합니다.

> ⚠️ **"내부용이라 괜찮아요" / "이번 한 번만" 은 없습니다.** 키·비밀번호·인증정보는 예외 없이
> 코드 밖으로 뺍니다. 보안에 "이번 한 번만"이라는 판단이 사고의 시작입니다.

### 올바른 방법: 키를 코드 밖으로

**① `application.properties`에 두고, 값은 환경변수에서 주입**

```properties
# 값 자체를 적지 않고, 환경변수 OPENAPI_KEY 에서 읽어 옵니다.
external.api.service-key=${OPENAPI_KEY}
```

```java
@Value("${external.api.service-key}")
private String serviceKey;   // 실행 시 환경변수 값이 들어옴
```

실행할 때 환경변수로 키를 넘깁니다(터미널/서버 설정에만 존재, 코드엔 없음):

```powershell
# PowerShell (현재 세션에만 설정)
$env:OPENAPI_KEY = "발급받은_키"
.\gradlew.bat bootRun
```

**② 키가 든 파일은 반드시 `.gitignore` 에** (예: `application-secret.properties`)
→ 커밋 대상에서 빠져 깃허브에 절대 안 올라갑니다. 우리 샘플의 `.gitignore`도 확인해 보세요.

| 방법 | 안전도 | 비고 |
|---|---|---|
| 소스에 하드코딩 | 🔴 최악 | 절대 금지 |
| `application.properties`에 평문으로 적고 커밋 | 🟠 나쁨 | 결국 깃에 올라감 |
| **환경변수 주입 `${...}`** | 🟢 기본 | 신입은 이것부터 |
| Vault·AWS Secrets Manager 등 비밀관리 도구 | 🟢 상용 | 규모 커지면 도입 |

> 🏢 **eGovFrame 연결 포인트** — 공공 SI에서도 DB 비밀번호·연계 API 키를 소스에 박지 않고
> `globals.properties`나 서버 환경설정(JNDI, 환경변수)으로 분리하는 것이 원칙입니다.
> 프레임워크는 달라도 **"비밀은 코드 밖에"** 라는 원칙은 똑같습니다.

## 5. 실무로 확장: 공공데이터포털(data.go.kr) 연동

회사에서 실제로 가장 많이 붙이는 외부 API입니다. 우리 샘플과 **구조가 똑같고**, 딱 두 가지만 다릅니다:
**① API 키(`serviceKey`)를 매 요청에 붙인다, ② 응답이 XML인 경우가 있다.**

```java
// data.go.kr 예: 한국천문연구원 특일 정보(공휴일) 조회 — 구조는 우리 ForecastClient와 동일
public HolidayResponse getHolidays(int year, int month) {
    return dataGoKrRestClient.get()
        .uri(uriBuilder -> uriBuilder
            .path("/B090041/openapi/service/SpcdeInfoService/getRestDeInfo")
            .queryParam("serviceKey", serviceKey)   // ← 발급받은 키 (환경변수에서 주입!)
            .queryParam("solYear", year)
            .queryParam("solMonth", String.format("%02d", month))
            .queryParam("_type", "json")            // JSON으로 달라고 요청 (기본은 XML)
            .build())
        .retrieve()
        .body(HolidayResponse.class);
}
```

**연동 순서 요약:**
1. data.go.kr 회원가입 → 원하는 API "활용신청" → **일반 인증키(serviceKey) 발급**
2. 키를 **환경변수/설정으로** 넣기 (4절 방식 — 절대 소스에 박지 않기)
3. 나머지는 우리 샘플과 동일: RestClient로 호출 → DTO로 받기 → 화면에 표시

> ⚠️ **함정 — serviceKey 이중 인코딩.** data.go.kr 키에는 `+`, `=`, `%` 가 들어 있습니다.
> 포털이 "Encoding 키"와 "Decoding 키" 두 가지를 주는데, `queryParam(...)`은 값을 **자동으로 인코딩**하므로
> **Decoding 키(원본)** 를 넣어야 합니다. Encoding 키를 넣으면 한 번 더 인코딩돼(`%2B%2B`) 인증에 실패합니다.
> "signature 불일치/인증 오류"가 나면 십중팔구 이 문제입니다.

> ⚠️ **함정 — JSON/XML.** 일부 공공 API는 `_type=json`을 지원하지만, **XML만 주는 것도 많습니다.**
> XML이면 DTO에 Jackson XML(`jackson-dataformat-xml`)을 쓰거나 XML 파서로 받아야 합니다.
> 신청 전에 그 API의 응답 포맷을 문서에서 꼭 확인하세요.

## 6. 그 밖에 챙길 것 (체크리스트)

| 항목 | 왜 |
|---|---|
| **호출 한도(Rate limit)** | 무료 API는 하루/분당 횟수 제한이 있음. 초과 시 429. 화면마다 매번 부르지 말고 필요시 **캐시** |
| **개인정보 최소 전송** | 외부에 보내는 요청에 불필요한 개인정보(이름·연락처)를 담지 말 것. 목적에 필요한 값만 |
| **로그에 키·개인정보 금지** | 디버그 로그(2절)에 요청 URL이 찍히면 그 안의 `serviceKey`도 찍힘. 운영 로그 레벨 주의 |
| **사내망 프록시** | 회사 망은 외부 인터넷을 프록시로만 허용하기도 함. 안 되면 프록시 설정 필요(인프라팀 문의) |
| **약관·라이선스** | 무료라도 이용약관·출처표기 의무가 있을 수 있음. 상용 서비스면 특히 확인 |

## 정리

- 외부 호출은 **반드시 실패한다고 가정**하고, 실패를 **종류별로**(입력/상대서버/네트워크) 나눠 처리합니다.
- **타임아웃은 필수** — 없으면 외부 장애가 우리 서버로 전염됩니다.
- **API 키는 절대 소스에 박지 않습니다.** 환경변수/설정으로 분리하고 `.gitignore`로 커밋을 막습니다.
  실수로 커밋했다면 **즉시 폐기·재발급**하세요.
- 이 원리로 회사의 **공공데이터포털** 같은 키 기반 API도 그대로 붙일 수 있습니다.

---

### 이 모듈을 마치며

DB(intro-jpa) → REST API 만들기(intro-api) → **외부 API 쓰기(weather)** 로,
스프링부트가 데이터를 다루는 세 가지 방향(내 DB / 남에게 주기 / 남에게 받기)을 모두 경험했습니다.
이제 여러분은 "데이터가 어디에 있든" 가져와 화면에 올릴 수 있습니다.

> **검증 정보(신뢰 수준 고지)** — 이 모듈의 `weather` 샘플은 실제로 빌드·실행하여
> Open-Meteo API를 호출하고, 정상/없는도시/한글입력 케이스를 모두 브라우저로 확인했습니다(스크린샷은 실측).
> 반면 **5절의 data.go.kr 코드는 개념 예시**입니다(개인 API 키가 필요해 이 자료에서는 실행 검증하지
> 않았습니다). 실제 연동 시 해당 API의 최신 명세를 반드시 공식 문서에서 확인하세요.
