# 05. (심화) 리프레시 토큰 적용하기 — auth-jwt-refresh 만들기

> **이 문서에서 배우는 것**
> - 액세스 토큰 하나만 쓰는 방식의 **딜레마**와 리프레시 토큰이 푸는 문제
> - **액세스 토큰(JWT, 무상태)** + **리프레시 토큰(DB 저장, 폐기 가능)** 의 역할 분담
> - **회전(rotation)** 과 **재사용 감지**, 그리고 **강제 로그아웃**
> - 완성본: [샘플/auth-jwt-refresh](./샘플/auth-jwt-refresh/)

> 이 문서는 [03. JWT 토큰 방식](./03_JWT_토큰_방식.md)의 **후속 심화편**입니다. 03을 먼저 보세요.

---

## 1. 왜 토큰을 두 개로 나누는가 — 딜레마에서 출발

03번 샘플(auth-jwt)의 액세스 토큰에는 근본적인 딜레마가 있었습니다.

| 만료를 **길게** 하면 | 만료를 **짧게** 하면 |
|---|---|
| 편하지만, **탈취당하면 오래 악용**됨 | 안전하지만, 사용자가 **자주 재로그인**해야 함 |
| 서버가 무상태라 **강제 로그아웃도 어려움** | 매번 아이디/비밀번호 입력은 최악의 UX |

두 마리 토끼를 잡는 방법이 **토큰을 두 개로 나누는 것**입니다.

| 토큰 | 정체 | 수명 | 저장 위치 | 역할 |
|---|---|---|---|---|
| **액세스 토큰** | JWT(서명됨) | **짧음**(데모 60초) | 클라이언트만 | 실제 API 호출에 사용 |
| **리프레시 토큰** | 임의 문자열(UUID) | **긺**(데모 14일) | **서버 DB** + 클라이언트 | 액세스 토큰을 재발급받는 열쇠 |

> 💡 **핵심 아이디어**: 자주 쓰는 액세스 토큰은 **짧게(탈취 대비)**, 가끔 쓰는 리프레시 토큰은 **길게**.
> 액세스 토큰이 만료되면 리프레시 토큰으로 **재로그인 없이** 새 액세스 토큰을 받아 옵니다.

## 2. 왜 리프레시 토큰만 DB에 저장하는가

03 문서에서 JWT의 약점으로 **"한번 발급하면 만료 전까지 강제로 못 막는다"** 를 배웠습니다.
리프레시 토큰을 **서버 DB에 저장**하면 이 약점을 보완할 수 있습니다.

- 서버가 "누구에게 어떤 리프레시 토큰을 줬는지" 알고 있으므로 → **DB에서 지우면 즉시 무효**(강제 로그아웃).
- 그래서 리프레시 토큰은 굳이 JWT로 만들 필요가 없습니다. 어차피 DB와 대조할 거라 **예측 불가능한 임의 문자열(UUID)** 이면 충분합니다.

> 💡 이 구조는 완전한 무상태가 아닙니다. **무상태(액세스) + 서버 상태(리프레시)** 의 **의도적 절충**입니다.
> "무상태의 확장성"을 대부분 누리면서, "서버 통제(강제 로그아웃)"를 리프레시 토큰에서만 확보합니다.

## 3. 리프레시 토큰 저장소 — 엔티티와 서비스

DB에 저장할 `RefreshToken` 엔티티입니다. (member 테이블에 더해 refresh_token 테이블이 생깁니다)

```java
@Entity @Table(name = "refresh_token")
public class RefreshToken {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY) private Long id;
    @Column(nullable=false, unique=true) private String token;    // 임의 UUID
    @Column(nullable=false) private String username;              // 토큰 주인
    @Column(nullable=false) private Instant expiresAt;            // 만료 시각
    public boolean isExpired() { return expiresAt.isBefore(Instant.now()); }
    // 생성자·getter 생략
}
```

발급·검증·회전·폐기를 담당하는 `RefreshTokenService` 의 핵심:

```java
// 발급: 임의 토큰을 만들어 DB에 저장
public String issue(String username) {
    String token = UUID.randomUUID().toString();
    Instant expiresAt = Instant.now().plus(validityDays, ChronoUnit.DAYS);
    refreshTokenRepository.save(new RefreshToken(token, username, expiresAt));
    return token;
}

// 검증: DB에 있고 만료 안 됐는지 확인
public RefreshToken validateAndGet(String token) {
    RefreshToken rt = refreshTokenRepository.findByToken(token)
        .orElseThrow(() -> new IllegalArgumentException("유효하지 않은 리프레시 토큰입니다."));
    if (rt.isExpired()) { refreshTokenRepository.delete(rt); throw new IllegalArgumentException("만료됨"); }
    return rt;
}

// 회전: 옛 토큰 폐기 + 새 토큰 발급
public String rotate(String oldToken, String username) {
    refreshTokenRepository.deleteByToken(oldToken);
    return issue(username);
}

// 폐기(로그아웃)
public void delete(String token) { refreshTokenRepository.deleteByToken(token); }
```

## 4. 4개의 엔드포인트 — login / refresh / logout

03과 비교해 **로그인 응답이 토큰 2개**가 되고, **refresh·logout** 이 추가됩니다.
(액세스 토큰 검증 필터·SecurityConfig 는 03과 동일합니다.)

```java
// 로그인: 액세스 + 리프레시 함께 발급
@PostMapping("/login")
public ResponseEntity<?> login(@RequestBody LoginRequest req) {
    Authentication auth = authenticationManager.authenticate(
        new UsernamePasswordAuthenticationToken(req.username(), req.password()));
    String role = auth.getAuthorities().iterator().next().getAuthority().replace("ROLE_", "");
    String access  = tokenProvider.createAccessToken(auth.getName(), role);
    String refresh = refreshTokenService.issue(auth.getName());
    return ResponseEntity.ok(new TokenResponse(access, refresh, role)); // 토큰 2개 반환
}

// 리프레시: 유효한 리프레시 토큰 → 새 액세스 + 새 리프레시(회전)
@PostMapping("/refresh")
public ResponseEntity<?> refresh(@RequestBody RefreshRequest req) {
    try {
        RefreshToken rt = refreshTokenService.validateAndGet(req.refreshToken());
        Member m = memberRepository.findByUsername(rt.getUsername()).orElseThrow(...);
        String newAccess  = tokenProvider.createAccessToken(m.getUsername(), m.getRole().name());
        String newRefresh = refreshTokenService.rotate(req.refreshToken(), m.getUsername());
        return ResponseEntity.ok(new TokenResponse(newAccess, newRefresh, m.getRole().name()));
    } catch (IllegalArgumentException e) {
        return ResponseEntity.status(401).body(new MessageResponse(e.getMessage()));
    }
}

// 로그아웃: 리프레시 토큰 폐기
@PostMapping("/logout")
public ResponseEntity<?> logout(@RequestBody RefreshRequest req) {
    refreshTokenService.delete(req.refreshToken());
    return ResponseEntity.ok(new MessageResponse("로그아웃되었습니다."));
}
```

> ⚠️ **함정 — refresh·logout 경로도 permitAll 이어야 합니다.**
> 이 두 요청은 **만료됐을 수도 있는** 상황에서 호출됩니다. 액세스 토큰을 요구하면 안 되므로
> `/api/auth/**` 를 `permitAll()` 로 열어 둡니다. 대신 리프레시 토큰 자체가 신분 증명 역할을 합니다.

> 💡 **왜 새 액세스 토큰을 만들 때 회원을 다시 조회할까?**
> 리프레시 시점에 관리자가 권한을 바꿨을 수도 있습니다. 토큰에 든 옛 권한이 아니라
> **DB의 최신 권한**으로 새 토큰을 발급하는 것이 안전합니다.

## 5. 회전(rotation)과 재사용 감지

**회전**이란 리프레시할 때마다 리프레시 토큰도 **새 것으로 갈아 주는** 것입니다. 왜 할까요?

```
로그인      → 리프레시토큰 A 발급 (DB: [A])
리프레시(A) → A 폐기 + B 발급     (DB: [B])   ← A는 이제 DB에 없음
리프레시(A) → DB에 A 없음 → 401  ← 이미 쓴 옛 토큰이 다시 오면 "탈취 의심"으로 거부
```

- 유출된 옛 토큰은 **다음 리프레시 순간 무용지물**이 됩니다.
- 옛 토큰이 재등장하면 **재사용 공격**을 감지할 수 있습니다(이 샘플은 감지 후 거부까지, 실무에선 "해당 사용자의 전체 토큰 폐기"까지 확장하기도 함).

> ⚠️ **jti(토큰 고유 ID)**: 액세스 토큰을 같은 "초"에 두 번 발급하면 `iat`/`exp` 가 같아 **똑같은 토큰**이 나옵니다.
> 이 샘플은 액세스 토큰에 `jti`(임의 UUID)를 넣어 **매번 다른 토큰**이 되게 했습니다.
> (`Jwts.builder().id(UUID.randomUUID().toString())`)

## 6. 실행하고 눈으로 확인하기

```bash
cd Spring/Security/샘플/auth-jwt-refresh
./gradlew bootRun
```

### 방법 A — 데모 페이지

http://localhost:8080 접속 후 순서대로:

1. **로그인** → 액세스·리프레시 토큰 2개가 상단에 표시됨
2. **/api/me** → 200 (액세스 토큰이 아직 살아 있음)
3. **60초 기다린 뒤 다시 /api/me** → **401**(액세스 토큰 만료)
4. **리프레시** → 새 토큰 2개 발급, 상단 토큰 값이 바뀜 → 다시 /api/me 하면 200
5. **로그아웃** → 리프레시 토큰 폐기. 이후 **리프레시를 또 누르면 401**

![리프레시 데모](./images/jwt_refresh_demo.png)

### 방법 B — 명령줄

```bash
# 1) 로그인 → 토큰 2개
curl -s -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" -d '{"username":"admin","password":"admin123"}'
# → {"accessToken":"eyJ...","refreshToken":"58ac3263-...","role":"ADMIN"}

# 2) 리프레시 → 새 토큰 2개 (리프레시 토큰도 회전됨)
curl -s -X POST http://localhost:8080/api/auth/refresh \
  -H "Content-Type: application/json" -d '{"refreshToken":"58ac3263-..."}'

# 3) 로그아웃 → 리프레시 토큰 폐기
curl -s -X POST http://localhost:8080/api/auth/logout \
  -H "Content-Type: application/json" -d '{"refreshToken":"<현재 리프레시 토큰>"}'
```

### 실제 검증 결과 (이 문서 작성 시 전부 실행 확인)

| 상황 | 요청 | 결과 |
|---|---|---|
| 로그인 | `POST /api/auth/login` | 200, `accessToken`+`refreshToken` 발급 |
| 액세스로 호출 | `GET /api/me` (admin) | 200 |
| 리프레시 | `POST /api/auth/refresh` | 200, **새 액세스≠옛 액세스, 새 리프레시≠옛 리프레시** |
| 회전된 옛 리프레시 재사용 | `POST /api/auth/refresh` (옛 토큰) | **401**(DB에 없음) |
| 로그아웃 후 리프레시 | `POST /api/auth/refresh` | **401**(폐기됨) |
| 위조 리프레시 | `POST /api/auth/refresh` | **401** |
| USER의 관리자 API | `GET /api/admin/dashboard` | **403** |

## 7. 세 샘플 비교 & 한계

| 샘플 | 토큰 | 강제 로그아웃 | 서버 상태 |
|---|---|---|---|
| auth-session | (세션) | 쉬움(세션 삭제) | 세션 보관 |
| auth-jwt | 액세스만 | **어려움** | 없음(완전 무상태) |
| **auth-jwt-refresh** | 액세스+리프레시 | **가능**(리프레시 폐기) | 리프레시 토큰만 |

> ⚠️ **이 샘플이 다루지 않는 것(실무 확장 포인트)**
> - **HttpOnly 쿠키 저장**: 리프레시 토큰을 JS가 못 읽는 쿠키에 담아 XSS를 방어(대신 CSRF 재고려).
> - **동시 세션/기기 관리**: 사용자별 여러 리프레시 토큰, "모든 기기 로그아웃"(`deleteByUsername`).
> - **만료 토큰 청소**: DB에 쌓인 만료 리프레시 토큰을 주기적으로 삭제하는 배치.
> - **재사용 감지 대응 강화**: 옛 토큰 재등장 시 해당 사용자 전체 토큰 폐기.
>
> 이 샘플은 **개념을 가장 명확히 보여 주는 최소 구조**에 집중했습니다. 실제 서비스 적용 시 위 항목과
> 조직 보안 정책을 함께 검토하세요.
