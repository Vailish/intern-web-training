# 04. 세션 vs JWT — 언제 무엇을 쓸까

> **이 문서에서 배우는 것**
> - 두 방식의 **핵심 차이**를 한 표로 정리
> - **판단 기준**: 내 상황에서 무엇을 골라야 하나
> - 각 방식의 **보안 유의사항**(CSRF·XSS·토큰 저장·무효화)
> - 실무에서 흔한 **하이브리드**(리프레시 토큰) 개념

---

## 결론부터 (Bottom Line)

- **서버가 화면(HTML)까지 그리는 웹**이면 → **세션**. (회사 eGovFrame이 이쪽)
- **프론트/백이 분리된 API·모바일 앱·여러 서버로 확장**이면 → **JWT**.
- 둘은 **경쟁이 아니라 상황별 도구**입니다. 실무에선 **"짧은 JWT + 리프레시 토큰"** 같은 절충도 흔합니다.
- **어느 쪽이든 비밀번호 해시(BCrypt)와 HTTPS는 필수** — 이건 선택의 문제가 아닙니다.

---

## 1. 핵심 비교표

| 항목 | 세션 방식 | JWT 토큰 방식 |
|---|---|---|
| **상태 저장** | 서버가 로그인 상태를 **기억**(stateful) | 서버는 **기억 안 함**(stateless) |
| **신분 증명 수단** | 세션 ID를 담은 **쿠키(JSESSIONID)** | 서명된 **토큰**(Authorization 헤더) |
| **저장 위치** | 상태=서버 메모리 / ID=브라우저 쿠키 | 토큰=클라이언트(localStorage 등) |
| **서버 확장(여러 대)** | 세션 공유 필요(끈적임/Redis 등) → **번거로움** | 각 서버가 서명만 검증 → **쉬움** |
| **로그아웃/강제 만료** | 서버가 세션 삭제 → **즉시·간단** | 토큰 만료 전까지 유효 → **어려움**(별도 장치 필요) |
| **주요 보안 위협** | **CSRF** | **XSS**(토큰 탈취), 토큰 무효화 |
| **모바일/외부 연동** | 쿠키 기반이라 **불편** | 헤더 기반이라 **적합** |
| **구현 난이도** | Security 기본 기능으로 **쉬움** | 필터·토큰 발급을 **직접 구현** |

### 실무 관점 지표 (도입 판단용)

| 지표 | 세션 | JWT | 코멘트 |
|---|---|---|---|
| **도입 난이도/리스크** | 낮음 | 중간 | JWT는 만료·저장·무효화를 잘못 설계하면 보안 구멍이 됨 |
| **수평 확장성** | 낮음 | 높음 | 서버가 늘어날수록 JWT가 유리 |
| **운영 편의(강제 로그아웃 등)** | 높음 | 낮음 | 세션은 서버에서 바로 끊을 수 있음 |
| **다양한 클라이언트 대응** | 낮음 | 높음 | 웹+앱+외부시스템이면 JWT |

## 2. 판단 흐름

```
화면(HTML)을 서버가 직접 그리나요?
├─ 예  → 세션 방식 (02 문서). 가장 단순하고 안전한 기본값.
└─ 아니오(REST API만 제공)
      ├─ 클라이언트가 브라우저 하나뿐?      → 세션도 충분히 가능
      └─ 모바일 앱/외부 시스템/서버 여러 대? → JWT (03 문서)
```

> 💡 **"요즘 다 JWT 쓴다던데?"** — 프론트/백 분리가 유행이라 JWT를 많이 보게 될 뿐입니다.
> 서버가 화면까지 그리는 전통적 웹에서는 **세션이 여전히 더 단순하고 안전한 정답**인 경우가 많습니다.
> 도구를 상황에 맞추세요.

## 3. 보안 유의사항 — 방식마다 급소가 다르다

### 세션 방식의 급소: CSRF

- 쿠키는 브라우저가 **자동으로** 붙이기 때문에, 악성 사이트가 사용자의 로그인 상태를 도용해
  몰래 요청을 보낼 수 있습니다(CSRF).
- **대응**: Spring Security의 **CSRF 토큰**(기본 켜짐). 타임리프 `th:action` 이 자동 삽입. (02 문서 6절)

### JWT 방식의 급소: XSS와 토큰 저장

- 토큰을 **`localStorage`** 에 두면, 사이트에 **XSS**(악성 스크립트 삽입) 취약점이 있을 때
  스크립트가 토큰을 통째로 훔쳐 갈 수 있습니다.
- **대응**:
  - 입력값 검증·이스케이프로 **XSS 자체를 막는 것이 최우선**.
  - 토큰을 `HttpOnly` 쿠키에 담으면 JS가 못 읽어 XSS에 강하지만, 그러면 **다시 CSRF를 고려**해야 합니다.
    (쿠키 vs localStorage는 "완벽한 정답"이 없는 트레이드오프입니다.)
  - **만료 시간을 짧게** 잡아 탈취되어도 피해 창을 줄입니다.

### JWT의 무효화 문제

- 서버가 기억을 안 하므로, 유출된 토큰을 **만료 전에 강제로 못 쓰게 하기가 어렵습니다.**
- **대응**(개념): 서버에 **블랙리스트**를 두거나(그만큼 상태를 갖게 됨), 아래 리프레시 토큰 패턴을 씁니다.

> ⚠️ **공통 필수 — HTTPS**
> 세션 쿠키든 JWT든, 네트워크에서 가로채이면 끝입니다. **운영 환경은 반드시 HTTPS**.
> 그리고 어느 방식이든 **비밀번호는 BCrypt 해시**(01 문서 5절)로 저장합니다. 이건 선택이 아니라 기본입니다.

## 4. 실무의 절충 — Access Token + Refresh Token

JWT의 "짧게 쓰고 싶다 vs 자주 로그인하긴 싫다" 딜레마를 푸는 흔한 패턴입니다.

- **Access Token**: 실제 API 호출에 쓰는 JWT. **만료를 짧게**(예: 30분). 탈취돼도 곧 무효.
- **Refresh Token**: Access Token이 만료되면 **새로 발급받기 위한** 토큰. 만료가 김(예: 2주),
  서버 DB에 저장해 두고 **강제 무효화도 가능**하게 만듭니다.

```
로그인 → Access(30분) + Refresh(2주) 발급
API 호출 → Access 사용
Access 만료 → Refresh 로 새 Access 발급 (재로그인 없이)
로그아웃/유출 → 서버가 Refresh 를 폐기 → 더는 갱신 불가
```

> 💡 이 패턴은 "무상태의 장점"과 "서버 통제(무효화)"를 **일부 절충**한 것입니다.
> 03번 샘플(auth-jwt)은 개념을 단순하게 보려고 Access Token 하나만 다뤘고,
> **이 패턴을 실제로 구현한 것이 [05. 리프레시 토큰 적용](./05_리프레시_토큰_적용.md)** 의 샘플 **auth-jwt-refresh** 입니다.
> 회전(rotation)·재사용 감지·강제 로그아웃까지 실행/검증해 두었으니 이어서 보세요.

## 5. 🏢 우리 회사(eGovFrame)와 연결

- eGovFrame 웹 애플리케이션(JSP + Spring MVC)은 대부분 **세션 방식**으로 로그인합니다.
  → 여러분이 실무에서 가장 먼저 만나는 형태이며, **02 문서**가 그 축소판입니다.
- eGovFrame도 내부적으로 **Spring Security** 를 씁니다. 즉 여기서 익힌
  `SecurityFilterChain`·인증/인가·`UserDetailsService` 개념이 **그대로 이어집니다.**
- 최근에는 여기에 **모바일 앱이나 외부 연동용 REST API** 를 얹으면서 **JWT** 를 추가하는 사례가 늘고 있습니다.
  → 이때 **03 문서**의 무상태 설정(STATELESS, CSRF 끔, 커스텀 필터)이 필요합니다.
