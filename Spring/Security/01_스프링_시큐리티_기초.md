# 01. 스프링 시큐리티 기초 — 인증, 인가, 그리고 필터체인

> **이 문서에서 배우는 것**
> - **인증(Authentication)** 과 **인가(Authorization)** 의 차이
> - Spring Security가 요청을 가로채는 방식 — **시큐리티 필터체인**
> - 핵심 3인방: `SecurityFilterChain`, `PasswordEncoder`, `UserDetailsService`
> - 비밀번호를 **왜, 어떻게** 암호화(해시)하는가
> - 스프링부트 4(Security 7)에서 달라진 설정 문법

---

## 1. Spring Security는 무엇을 해 주는가

지금까지 만든 앱은 주소만 알면 누구나 아무 페이지에 들어갈 수 있었습니다.
실제 서비스는 그럴 수 없죠. "로그인한 사람만", "관리자만" 같은 규칙이 필요합니다.

**Spring Security** 는 이런 일을 대신해 주는 **보안 프레임워크**입니다. 직접 만들면
매우 어렵고 실수하기 쉬운 부분(비밀번호 처리, 세션 관리, CSRF 방어 등)을
검증된 방식으로 제공합니다.

의존성만 추가해도 **모든 요청에 로그인을 요구**하는 기본 보안이 즉시 켜집니다.
우리가 하는 일은 그 기본 위에 **"우리 서비스에 맞는 규칙"** 을 얹는 것입니다.

---

## 2. 인증 vs 인가 — 가장 먼저 구분해야 할 두 단어

두 단어는 늘 붙어 다니지만 **다른 일**입니다.

| 구분 | 영어 | 질문 | 예시 |
|---|---|---|---|
| **인증** | Authentication | "너 **누구**야?" | 아이디/비밀번호로 로그인 |
| **인가** | Authorization | "너 이거 **해도 돼**?" | 관리자만 관리자 페이지 접근 |

순서도 이 순서입니다. **먼저 누구인지 확인(인증)** 하고, **그 다음 권한을 검사(인가)** 합니다.
로그인은 했지만(인증 O) 권한이 없어서(인가 X) 거부되는 경우가 바로 `403 Forbidden` 입니다.

> 💡 로그인을 아예 안 했으면? 세션 방식은 **로그인 페이지로 보내고(302)**, API(JWT) 방식은
> **`401 Unauthorized`** 를 돌려줍니다. 권한이 없으면 둘 다 **`403 Forbidden`**.

---

## 3. 요청이 컨트롤러에 닿기 전에 — 시큐리티 필터체인

Spring Security의 동작 원리를 한 문장으로 요약하면 이렇습니다.

> **모든 HTTP 요청은 컨트롤러에 도착하기 전에, 줄지어 선 "필터(Filter)"들을 차례로 통과한다.**

이 필터들의 줄이 **시큐리티 필터체인(Security Filter Chain)** 입니다. 각 필터가 한 가지씩 검사합니다.

```
[브라우저 요청]
     │
     ▼
 ┌───────────────── 시큐리티 필터체인 ─────────────────┐
 │  ... → 로그인 처리 필터 → 인가 검사 필터 → ...       │
 └──────────────────────┬──────────────────────────────┘
                        ▼  (통과한 요청만)
                   [컨트롤러]
```

- **세션 방식**은 이 필터체인이 제공하는 **폼 로그인 필터**를 그대로 씁니다. (02 문서)
- **JWT 방식**은 우리가 만든 **JWT 검사 필터**를 이 줄에 끼워 넣습니다. (03 문서)

우리가 필터를 직접 다루는 일은 많지 않습니다. 대신 **"규칙"을 선언**하면 Security가 알아서 필터를 구성합니다.
그 규칙을 적는 곳이 다음 절의 `SecurityFilterChain` 빈입니다.

## 4. 핵심 3인방

두 샘플 모두 아래 세 가지가 뼈대입니다. 이름과 역할을 먼저 익혀 두세요.

### (1) `SecurityFilterChain` — 보안 규칙을 적는 곳

"어떤 URL은 아무나, 어떤 URL은 로그인해야, 어떤 URL은 관리자만"을 선언하는 설정 빈입니다.

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/", "/login", "/signup").permitAll() // 아무나
            .requestMatchers("/admin/**").hasRole("ADMIN")          // 관리자만
            .anyRequest().authenticated()                          // 나머지는 로그인 필요
        );
    return http.build();
}
```

- `permitAll()` : 로그인 없이 허용
- `authenticated()` : 로그인(인증)만 되어 있으면 허용
- `hasRole("ADMIN")` : `ROLE_ADMIN` 권한이 있어야 허용
- **규칙은 위에서부터 순서대로 매칭**됩니다. 넓은 규칙(`anyRequest`)은 항상 맨 아래에 둡니다.

### (2) `PasswordEncoder` — 비밀번호 암호화 도구

비밀번호는 **절대 원문으로 저장하지 않습니다.** `BCryptPasswordEncoder` 로 **단방향 해시**해서 저장합니다.

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

### (3) `UserDetailsService` — DB와 Security를 잇는 다리

로그인할 때 Security가 "이 아이디의 회원 정보를 줘"라고 호출하는 통로입니다.
우리는 아이디로 회원을 찾아 Security가 이해하는 형태(`UserDetails`)로 돌려주기만 하면 됩니다.
**비밀번호 비교는 Security가 알아서** 합니다.

```java
@Override
public UserDetails loadUserByUsername(String username) {
    Member member = memberRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("없는 회원: " + username));
    return User.builder()
            .username(member.getUsername())
            .password(member.getPassword())                 // 이미 해시된 값
            .authorities("ROLE_" + member.getRole().name()) // 예: ROLE_ADMIN
            .build();
}
```

> ⚠️ **`ROLE_` 접두사 규칙**
> 설정에서 `hasRole("ADMIN")` 이라고 쓰면 Security는 실제로는 **`ROLE_ADMIN`** 권한을 찾습니다.
> 그래서 권한을 만들 때 `"ROLE_" + 역할이름` 으로 접두사를 **직접 붙여** 줘야 합니다.
> (반대로 `hasAuthority("ROLE_ADMIN")` 은 접두사까지 정확히 적는 방식입니다.)

## 5. 비밀번호는 왜 "해시"하는가

만약 DB에 비밀번호를 원문(`user123`)으로 저장했다가 DB가 유출되면, 모든 회원의 비밀번호가
그대로 털립니다. 사람들은 여러 사이트에 같은 비밀번호를 쓰기 때문에 피해가 훨씬 커집니다.

그래서 **되돌릴 수 없는(단방향)** 함수로 바꿔서 저장합니다. 이것이 **해시**입니다.

```
"user123"  ──BCrypt──▶  "$2a$10$N9qo8uLOickgx2ZMRZoMy..."   ← 이 값이 DB에 저장됨
```

- **되돌리기 불가**: 해시값만으로는 원문 `user123` 을 알 수 없습니다.
- **로그인 검증 방법**: 사용자가 입력한 비밀번호를 **똑같이 해시해서**, DB의 해시값과 **비교**만 합니다.
- **BCrypt의 특징**: 같은 비밀번호라도 매번 다른 해시가 나오고(내부에 임의값 salt 포함),
  일부러 계산을 느리게 만들어 무차별 대입 공격을 어렵게 합니다.

> ⚠️ **해시 ≠ 암호화**
> 암호화(encryption)는 키가 있으면 **원문으로 되돌릴 수** 있습니다(양방향).
> 비밀번호는 되돌릴 필요가 전혀 없으므로 **되돌릴 수 없는 해시(hash)** 를 씁니다. 더 안전합니다.

우리 샘플의 회원가입 코드가 바로 이 일을 합니다.

```java
String encodedPassword = passwordEncoder.encode(rawPassword); // 해시해서 저장
memberRepository.save(new Member(username, encodedPassword, ...));
```

## 6. ⚠️ 스프링부트 4(Spring Security 7)에서 달라진 것

인터넷 예제를 그대로 따라 하면 컴파일이 안 되는 경우가 많습니다. Boot 버전별 차이를 알아 두세요.

> ⚠️ **함정 1 — `WebSecurityConfigurerAdapter` 는 이제 없습니다.**
> 아주 오래된 예제는 `class SecurityConfig extends WebSecurityConfigurerAdapter` 로 시작합니다.
> 이 방식은 Spring Security 6에서 이미 제거됐습니다. **`SecurityFilterChain` 빈을 반환**하는
> 지금 방식만 사용하세요.

> ⚠️ **함정 2 — 설정은 전부 "람다"로 써야 합니다.**
> Security 7에서는 `.and()` 로 이어 쓰던 옛날 문법이 사라졌습니다.
> `authorizeHttpRequests(auth -> auth. ...)` 처럼 **화살표 람다** 스타일만 됩니다.
> (이 모듈의 모든 예제가 람다 스타일입니다.)

> ⚠️ **함정 3 — 의존성 이름이 바뀌었습니다.**
> 웹 스타터는 `spring-boot-starter-web` 이 아니라 **`spring-boot-starter-webmvc`**,
> H2 콘솔은 별도 의존성 **`spring-boot-h2console`** 입니다. (Database 모듈에서 배운 그대로)

> ⚠️ **함정 4 — 메서드 이름이 조금씩 바뀌었습니다.**
> 예전 `antMatchers()` → 지금 **`requestMatchers()`**,
> `authorizeRequests()` → **`authorizeHttpRequests()`**.
> 검색해서 나온 코드가 안 되면 이 대응부터 의심하세요.
