# 04. 완전탐색과 그리디 — 다 해보기 vs 매 순간 최선

> **이 문서에서 배우는 것**
> - **완전탐색(브루트포스)**: 모든 경우를 다 시도 — 언제 이게 "정답"인가
> - **순열·조합**을 재귀로 만드는 법 (경우의 수 생성의 기본기)
> - **백트래킹**: 가망 없는 가지를 쳐내며 탐색을 줄이는 기술 (N-Queens)
> - **그리디**: 매 순간 최선을 골라 전체 최적에 도달 — 언제 통하고 언제 안 통하나

이 두 접근은 정반대입니다. 완전탐색은 "확실하지만 느릴 수 있고", 그리디는 "빠르지만 항상 옳지는 않습니다."
**입력 크기 N**이 둘 중 무엇을 쓸지 결정합니다. ([01번 복잡도 표](./01_기초_시간복잡도와_입출력.md)를 기억하세요.)

---

## 1. 완전탐색 — 모든 경우를 다 해본다

가장 정직한 방법. **"모든 경우의 수", "가능한 모든 조합"** 이라는 말이 보이고 **N이 작으면(대체로 20 이하)** 먼저 의심합니다.

> 💡 완전탐색이 매력적인 이유: **틀릴 수가 없습니다.** 다 해보니까요.
> 문제는 오직 시간뿐. N이 작으면 "복잡한 최적해를 고민하기 전에 그냥 다 해보는 게" 정답인 경우가 많습니다.

가장 단순한 형태는 중첩 반복문입니다.

```java
// 세 수를 뽑아 합이 target이 되는 경우 (N이 작을 때 O(N^3))
for (int i = 0; i < n; i++)
    for (int j = i + 1; j < n; j++)
        for (int k = j + 1; k < n; k++)
            if (a[i] + a[j] + a[k] == target) count++;
```

하지만 "몇 개를 뽑을지"가 가변이면 반복문으로는 한계가 있습니다. → **재귀**로 경우의 수를 생성합니다.

---

## 2. 순열과 조합 — 경우의 수 생성의 기본기

| 구분 | 의미 | 예 (1,2,3에서 2개) |
|---|---|---|
| **순열** (Permutation) | 순서가 **의미 있음** | (1,2) ≠ (2,1) → 6가지 |
| **조합** (Combination) | 순서 **무관** | (1,2) = (2,1) → 3가지 |

### 조합 — "선택할지 말지" 재귀

```java
// arr에서 r개를 고르는 모든 조합
static void combination(int[] arr, boolean[] visited, int start, int r) {
    if (r == 0) {                         // r개를 다 골랐으면 출력
        print(arr, visited);
        return;
    }
    for (int i = start; i < arr.length; i++) {
        visited[i] = true;
        combination(arr, visited, i + 1, r - 1);   // i+1부터 → 뒤로만 진행(중복 방지)
        visited[i] = false;                          // 되돌리기(백트래킹)
    }
}
```

### 순열 — 순서가 중요하니 매번 처음부터

순열은 "이미 쓴 것"만 빼고 매번 전체를 후보로 둡니다. 자세한 코드는 [샘플/src/Ex09_Permutation.java](./샘플/src/Ex09_Permutation.java).

> 📌 **핵심 패턴 = 선택 → 재귀 → 선택 취소.** 이 "취소(되돌리기)"가 바로 백트래킹의 뼈대입니다.
> `visited[i] = true; ...재귀...; visited[i] = false;` 이 3줄 구조를 손에 익히세요.

---

## 3. 백트래킹 — 가망 없는 길은 일찍 포기

완전탐색을 하되, **"이 방향은 어차피 답이 안 된다"** 싶으면 더 내려가지 않고 되돌아옵니다(가지치기, pruning).
경우의 수가 폭발하는 문제에서 실제로 탐색해야 할 양을 극적으로 줄입니다.

### 대표 예: N-Queens

> N×N 체스판에 퀸 N개를, 서로 공격하지 못하게 놓는 경우의 수.
> 퀸은 같은 행·열·대각선을 공격합니다.

한 행에 하나씩 놓아가되, **놓기 전에 "이 자리가 안전한가"를 검사**하고 안전할 때만 다음 행으로 내려갑니다.
안전하지 않으면 그 가지는 즉시 버립니다 — 이게 가지치기입니다.

```java
static int solve(int row, int n, int[] col) {
    if (row == n) return 1;               // N개를 다 놓음 → 성공 1가지
    int count = 0;
    for (int c = 0; c < n; c++) {
        if (isSafe(row, c, col)) {        // 안전할 때만 진행 (가지치기)
            col[row] = c;
            count += solve(row + 1, n, col);
            // col[row] 되돌리기는 다음 c에서 덮어쓰므로 생략 가능
        }
    }
    return count;
}
```

전체 실행 예제: [샘플/src/Ex10_Backtracking_NQueens.java](./샘플/src/Ex10_Backtracking_NQueens.java)

> ⚠️ 백트래킹도 최악엔 지수 시간입니다. **N이 크면** 백트래킹으로도 안 되고 DP나 수학이 필요합니다.
> "N ≤ 15 정도, 모든 배치를 따져야 함" → 백트래킹의 전형적 신호.

---

## 4. 그리디 — 매 순간 최선을 고른다

전체를 다 보지 않고, **각 단계에서 가장 좋아 보이는 선택**을 하며 앞으로만 나아갑니다. 빠릅니다(대개 정렬 O(N log N) + 한 번 훑기).

### 대표 예 1: 거스름돈 (동전 최소 개수)

큰 동전부터 최대한 쓰면 개수가 최소가 됩니다. (**단, 동전 체계가 배수 관계일 때만** — 아래 함정 참고)

```java
int[] coins = {500, 100, 50, 10};   // 큰 것부터 정렬돼 있다고 가정
int change = 1260, count = 0;
for (int coin : coins) {
    count += change / coin;         // 이 동전을 최대한 사용
    change %= coin;                 // 남은 거스름돈
}
// count = 2(500) + 2(100) + 1(50) + 1(10) = 6개
```

### 대표 예 2: 회의실 배정 (가장 빨리 끝나는 것부터)

> 겹치지 않게 최대한 많은 회의를 배정하려면? → **끝나는 시간이 빠른 순**으로 정렬해 욕심내어 담습니다.

이 "무엇을 기준으로 정렬하면 그리디가 성립하는가"를 찾는 게 그리디 문제의 핵심입니다.
전체 예제: [샘플/src/Ex11_Greedy.java](./샘플/src/Ex11_Greedy.java)

### ⚠️ 그리디의 함정 — 항상 최적은 아니다

그리디는 **틀릴 수 있습니다.** 매 순간 최선이 전체 최선이라는 보장은 없습니다.

```
동전이 {1, 4, 5}이고 8원을 거슬러야 한다면?
 - 그리디(큰 것부터): 5 + 1 + 1 + 1 = 4개   ❌
 - 최적:              4 + 4         = 2개   ✅
```

이 경우 그리디는 틀립니다. **왜 그리디가 성립하는지 근거(정렬 기준의 정당성)를 설명할 수 없다면 함부로 쓰면 안 됩니다.**
확신이 없고 N이 작다면 → 완전탐색/DP가 안전합니다.

> 💡 판단법: **"큰 것/빠른 것부터 골라도 손해가 없다"는 걸 논리로 보일 수 있으면 그리디.**
> 아니면 DP([06번](./06_동적계획법_DP.md))를 의심하세요. 실제로 거스름돈·배낭 문제는 DP의 단골입니다.

---

## 완전탐색 vs 그리디 vs DP — 언제 무엇을?

| 상황 | 접근 | 근거 |
|---|---|---|
| N이 작다(≤ 20~25), 확실한 답이 필요 | **완전탐색/백트래킹** | 다 해봐도 시간 안에 됨 |
| 정렬 기준의 최적성을 논증할 수 있음 | **그리디** | 빠름, 하지만 증명이 전제 |
| 큰 N + 겹치는 부분문제 + 최적화 | **DP** | 완전탐색의 중복 계산 제거 |

---

## 정리

- 완전탐색은 **N이 작을 때의 확실한 정답**. "모든 경우"라는 말 + 작은 N을 보면 먼저 떠올린다.
- 경우의 수 생성 = **선택 → 재귀 → 선택 취소** 패턴. 조합은 `start` 인덱스로 중복을 막는다.
- 백트래킹은 완전탐색 + **가지치기**. 안 될 가지를 일찍 버려 탐색량을 줄인다.
- 그리디는 빠르지만 **틀릴 수 있다.** 정렬 기준의 정당성을 설명할 수 없으면 쓰지 말 것.
