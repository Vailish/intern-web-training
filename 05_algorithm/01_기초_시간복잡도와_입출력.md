# 01. 시간복잡도와 입출력 — 코테의 절반은 여기서 갈린다

> **이 문서에서 배우는 것**
> - **시간복잡도(Big-O)** 가 무엇이고, 왜 코테 합격의 핵심인지
> - 입력 크기 N을 보고 **"어떤 복잡도까지 허용되는지" 역산**하는 법
> - 자바에서 **시간초과를 부르는 입출력**과, 이를 막는 표준 템플릿
> - `String` 덧셈이 왜 위험한지 — 일부러 느리게 만들어 봅니다

코테에서 떨어지는 가장 흔한 이유는 "답을 몰라서"가 아니라 **"답은 맞는데 느려서"** 입니다.
이 문서는 그 벽을 넘는 문서입니다. 다른 어떤 알고리즘보다 먼저 몸에 익히세요.

---

## 1. 시간복잡도(Big-O)란?

**입력 크기 N이 커질 때, 연산 횟수가 얼마나 빠르게 늘어나는가**를 나타내는 표기법입니다.
정확한 시간(초)이 아니라 **증가 추세**를 봅니다.

```java
// O(1) — 입력 크기와 무관하게 항상 한 번
int first = arr[0];

// O(N) — 배열을 한 번 훑음
for (int i = 0; i < n; i++) sum += arr[i];

// O(N^2) — 이중 반복. N이 2배면 시간은 4배
for (int i = 0; i < n; i++)
    for (int j = 0; j < n; j++)
        if (arr[i] + arr[j] == target) count++;
```

핵심 규칙 세 가지:

| 규칙 | 예 | 결과 |
|---|---|---|
| 상수는 무시 | `3N + 5` | O(N) |
| 가장 큰 항만 남김 | `N^2 + N` | O(N^2) |
| 중첩은 곱셈 | 반복 N × 반복 N | O(N^2) |

> 📌 왜 상수를 무시할까요? N=1,000,000일 때 `2N`이든 `5N`이든 `N^2`(1조!) 앞에서는 사실상 같습니다.
> Big-O는 **"규모가 커질 때 누가 이기냐"** 만 봅니다.

### 복잡도의 크기 순서 (작을수록 빠름)

```
O(1) < O(log N) < O(N) < O(N log N) < O(N^2) < O(2^N) < O(N!)
       └ 이진탐색        └ 정렬        └ 이중반복  └ 부분집합  └ 순열
```

## 2. 가장 중요한 표 — N을 보고 복잡도를 역산하라

코테는 보통 **시간 제한 1~2초**를 줍니다. 자바는 대략 **1초에 1억(10^8) 번** 연산한다고 잡습니다.
그러면 문제의 입력 크기 N만 봐도 **허용되는 복잡도가 정해집니다.**

| 입력 크기 N | 최대 허용 복잡도 | 이런 접근이 가능 |
|---|---|---|
| N ≤ 11 | O(N!) ≈ 4천만 | 모든 순열 완전탐색 |
| N ≤ 25 | O(2^N) ≈ 3천만 | 모든 부분집합 |
| N ≤ 500 | O(N^3) ≈ 1.2억 | 3중 반복, 플로이드-워셜 |
| N ≤ 3,000 | O(N^2) ≈ 900만 | 2중 반복, 기본 DP |
| N ≤ 100,000 | O(N log N) ≈ 170만 | 정렬, 이진탐색, 힙 |
| N ≤ 10,000,000 | O(N) | 한 번만 훑기 |
| N이 10^12처럼 거대 | O(log N) 또는 O(1) | 이진탐색/수학 공식 |

**사용법 (반대로 읽습니다):**

> 문제에 `N ≤ 1,000`이라고 쓰여 있다 → O(N^2)까지는 안전 → **이중 반복으로 풀어도 된다.**
> 문제에 `N ≤ 100,000`이라고 쓰여 있다 → O(N^2)면 100억 → **시간초과.** O(N log N) 이하로 설계해야 한다.

이 한 줄이 **"어떤 알고리즘을 쓸지"의 첫 단서**입니다. 문제를 읽으면 제일 먼저 N의 범위를 찾으세요.

## 3. 공간복잡도 — 메모리초과도 탈락이다

시간과 같은 논리로, **메모리 사용량**도 N에 따라 늘어납니다. 코테는 보통 **256MB** 제한을 줍니다.

- `int` 하나 = 4바이트. `int[1000][1000]` = 400만 개 × 4 = 약 16MB → OK
- `int[100000][100000]` = 100억 개 × 4 = 40GB → **메모리초과.** 2차원 배열 크기를 항상 계산하세요.
- `boolean[]` (1바이트)가 `int[]`(4바이트)보다 4배 가볍습니다. 방문 체크는 `boolean[]`로.

> ⚠️ 재귀가 너무 깊으면 **StackOverflowError** — 이것도 일종의 공간 문제입니다.
> 재귀 깊이가 수십만이 될 것 같으면 반복문/명시적 스택으로 바꾸세요. ([05번 DFS 문서](./05_그래프탐색_DFS_BFS.md) 참고)

## 4. 자바의 함정 ① — 느린 입출력

**알고리즘이 맞아도 입출력이 느리면 시간초과가 납니다.** 자바 초심자가 가장 많이 당하는 함정입니다.

### 입력: `Scanner`는 느리다 → `BufferedReader`

```java
// ❌ 느림 — 입력이 수십만 줄이면 이것만으로 시간초과
Scanner sc = new Scanner(System.in);
int n = sc.nextInt();

// ✅ 빠름 — 표준 템플릿
BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
int n = Integer.parseInt(br.readLine());
```

한 줄에 여러 숫자가 있으면 `StringTokenizer`로 쪼갭니다.

```java
StringTokenizer st = new StringTokenizer(br.readLine());
int a = Integer.parseInt(st.nextToken());
int b = Integer.parseInt(st.nextToken());
```

### 출력: 반복 `System.out.println`도 느리다 → `StringBuilder`로 모아서 한 번에

```java
// ❌ 10만 번 반복 출력 → 느림
for (int x : answers) System.out.println(x);

// ✅ StringBuilder에 모아 마지막에 한 번만 출력
StringBuilder sb = new StringBuilder();
for (int x : answers) sb.append(x).append('\n');
System.out.print(sb);
```

### 표준 입출력 템플릿 (외워두면 편합니다)

전체 실행 예제는 [샘플/src/Ex02_FastIO.java](./샘플/src/Ex02_FastIO.java)에 있습니다.

```java
import java.io.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        StringBuilder sb = new StringBuilder();

        int n = Integer.parseInt(br.readLine());
        for (int i = 0; i < n; i++) {
            StringTokenizer st = new StringTokenizer(br.readLine());
            int a = Integer.parseInt(st.nextToken());
            int b = Integer.parseInt(st.nextToken());
            sb.append(a + b).append('\n');
        }
        System.out.print(sb);
    }
}
```

> 📌 `main`에 `throws IOException`을 붙이는 이유: `br.readLine()`이 검사 예외를 던지므로, 코테에서는
> 매번 try-catch 하기보다 `throws`로 넘기는 게 관례입니다.

## 5. 자바의 함정 ② — `String` 덧셈은 O(N^2)

문자열을 반복해서 `+`로 이어붙이면, 자바는 **매번 새 문자열을 통째로 복사**합니다.
N번 이어붙이면 1+2+3+...+N ≈ **O(N^2)** — 이것만으로 시간초과가 납니다.

```java
// ❌ 반복문 안에서 String +=  → O(N^2)
String result = "";
for (int i = 0; i < n; i++) result += i;   // 매번 전체 복사!

// ✅ StringBuilder → O(N)
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) sb.append(i);
String result = sb.toString();
```

`Ex02_FastIO.java`는 이 둘의 시간 차이도 실제로 재서 보여줍니다. `String +=`가 얼마나 극적으로 느려지는지 확인하세요.

> 💡 규칙: **반복문 안에서 문자열을 만든다 → 무조건 `StringBuilder`.** 예외 없습니다.

## 6. 함정 ③ — 자료형 오버플로

`int`의 최댓값은 약 21억(2^31-1 ≈ 2.1×10^9)입니다. 곱셈·누적합에서 이를 넘으면 **음수로 뒤집힙니다(오버플로).**

```java
int a = 100000, b = 100000;
int wrong = a * b;              // 100억 → int 범위 초과 → 쓰레기 값
long right = (long) a * b;      // ✅ 계산 전에 long으로 승격
```

> ⚠️ `long total = a * b;` 는 **틀렸습니다.** `a * b`가 이미 int로 계산된 뒤 long에 담기므로 늦었습니다.
> **곱하기 전에** 한쪽을 `(long)`으로 캐스팅해야 합니다. "합/곱이 21억을 넘을 수 있나?" 싶으면 `long`을 쓰세요.
