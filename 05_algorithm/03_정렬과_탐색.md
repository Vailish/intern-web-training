# 03. 정렬과 탐색 — 정렬은 무기, 이진탐색은 지렛대

> **이 문서에서 배우는 것**
> - 자바 기본 정렬 `Arrays.sort` / `Collections.sort` 와 **Comparator로 원하는 기준 정렬**
> - **이진탐색(Binary Search)**: 정렬된 데이터에서 O(log N)으로 찾기
> - **파라메트릭 서치**: "답 자체를 이진탐색"하는 강력한 발상 (코테 고득점 단골)

정렬은 그 자체로 답이 되기도 하고, 다른 알고리즘의 **전처리**로도 쓰입니다.
이진탐색은 "정렬돼 있다"는 조건만 있으면 탐색을 O(N)에서 O(log N)으로 끌어내리는 지렛대입니다.

---

## 1. 기본 정렬

### 원시 배열 / 리스트 정렬

```java
int[] arr = {5, 2, 8, 1};
Arrays.sort(arr);                     // 오름차순: [1, 2, 5, 8]

Integer[] boxed = {5, 2, 8, 1};
Arrays.sort(boxed, Collections.reverseOrder());   // 내림차순 (객체 배열만 가능)

List<Integer> list = new ArrayList<>(List.of(5, 2, 8, 1));
Collections.sort(list);               // 오름차순
list.sort(Comparator.reverseOrder()); // 내림차순
```

> ⚠️ **원시 타입 `int[]`는 내림차순 정렬이 없습니다.** `Arrays.sort(intArr, reverseOrder)`는 컴파일 에러입니다.
> 내림차순이 필요하면 ① `Integer[]`로 바꾸거나 ② 오름차순 정렬 후 뒤집거나 ③ 부호를 반대로 넣었다 빼세요.

정렬의 시간복잡도는 **O(N log N)**. N ≤ 100,000까지 넉넉히 감당합니다.

### Comparator — 내가 원하는 기준으로 정렬

객체나 다중 기준 정렬의 핵심. 코테에서 매우 자주 나옵니다.

```java
// 예: 학생을 "점수 내림차순, 점수 같으면 이름 오름차순"으로 정렬
record Student(String name, int score) {}
List<Student> students = new ArrayList<>(...);

students.sort(
    Comparator.comparingInt(Student::score).reversed()   // 1순위: 점수 내림차순
              .thenComparing(Student::name)              // 2순위: 이름 오름차순
);
```

핵심 메서드:

| 메서드 | 의미 |
|---|---|
| `Comparator.comparingInt(x -> x.field)` | 정수 필드 오름차순 |
| `.reversed()` | 방향 뒤집기(내림차순) |
| `.thenComparing(...)` | 앞 기준이 같을 때의 2차 기준 |

> 📌 직접 람다로 쓸 수도 있습니다: `list.sort((a, b) -> a.score() - b.score());` (오름차순).
> 단, **뺄셈 비교는 오버플로 위험**이 있으니 큰 수엔 `Integer.compare(a, b)`를 쓰세요.

## 2. 이진탐색 — 정렬돼 있으면 O(log N)

**정렬된** 배열에서 값을 찾을 때, 매번 탐색 범위를 절반으로 줄입니다.
N이 100만이어도 약 20번(log₂ 1,000,000 ≈ 20)이면 끝납니다.

```java
// 정렬된 arr에서 target의 인덱스를 찾음 (없으면 -1)
static int binarySearch(int[] arr, int target) {
    int lo = 0, hi = arr.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;    // (lo+hi)/2 대신 → 오버플로 방지
        if (arr[mid] == target) return mid;
        else if (arr[mid] < target) lo = mid + 1;   // 오른쪽 절반
        else hi = mid - 1;                            // 왼쪽 절반
    }
    return -1;
}
```

> 📌 `mid = lo + (hi - lo) / 2`로 쓰는 이유: `(lo + hi)`가 int 범위를 넘을 수 있어서입니다.
> 습관으로 이렇게 쓰세요.

### 자바 내장 이진탐색

직접 구현이 헷갈리면 내장 함수를 쓰되, **없을 때의 반환값 규칙**을 알아야 합니다.

```java
int[] arr = {1, 3, 5, 7, 9};
int idx = Arrays.binarySearch(arr, 5);   // 2 (찾음)
int miss = Arrays.binarySearch(arr, 4);  // 음수: -(들어갈 위치)-1
```

### lower/upper bound — "그 값 이상이 처음 나오는 위치"

중복이 있거나 "X 이상인 것의 개수" 같은 문제에선 위치를 찾는 이진탐색이 필요합니다.

## 3. 파라메트릭 서치 — "답을 이진탐색한다"

코테 중급의 벽이자 자주 나오는 강력한 발상입니다.

> **핵심 아이디어**: 최적화 문제("~할 수 있는 최댓값/최솟값은?")를
> **결정 문제("답이 X일 때 가능한가? Yes/No")** 로 바꾼 뒤, **X를 이진탐색**합니다.

**성립 조건**: X가 커질수록(작아질수록) 답이 Yes→No로 **단조롭게** 바뀔 때.

### 대표 예: 랜선 자르기 / 나무 자르기

> "길이가 제각각인 랜선들을 잘라서, 길이 X짜리를 K개 이상 만들고 싶다.
> 가능한 **최대 길이 X**는?"

- 만약 X=5로 K개를 만들 수 있다면, X=3, X=4도 당연히 가능(단조성!).
- X를 1씩 늘려보면 O(최대길이)라 시간초과 → **X를 이진탐색**하면 O(log(최대길이) × N).

```java
// "길이 mid로 자르면 총 몇 개가 나오나"를 세는 판정 함수
static long countPieces(int[] lengths, int cut) {
    long cnt = 0;
    for (int len : lengths) cnt += len / cut;   // 각 랜선에서 몇 개 나오는지
    return cnt;
}

// 답(자를 길이)을 이진탐색
static int maxCutLength(int[] lengths, int need) {
    int lo = 1, hi = 0;
    for (int len : lengths) hi = Math.max(hi, len);   // 답의 상한 = 가장 긴 랜선

    int answer = 0;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (countPieces(lengths, mid) >= need) {   // mid로 충분히 만들 수 있다면
            answer = mid;        // 답 후보로 저장하고
            lo = mid + 1;        // 더 길게(더 좋은 답) 시도
        } else {
            hi = mid - 1;        // 너무 길어서 개수 부족 → 줄임
        }
    }
    return answer;
}
```

전체 실행 예제: [샘플/src/Ex08_ParametricSearch.java](./샘플/src/Ex08_ParametricSearch.java)

> 💡 파라메트릭 서치의 신호: **"~할 수 있는 최대/최소"** + **답의 범위가 매우 큼(1억 이상)** + **답이 정해지면 가능/불가능 판정이 쉬움.**
> 이 세 개가 보이면 "답을 이진탐색"을 떠올리세요.

## 정리

- **정렬**은 O(N log N). Comparator로 다중 기준을 세운다. (`comparing` → `thenComparing`)
- **이진탐색**은 정렬 전제에서 O(log N). `mid = lo + (hi-lo)/2`로 오버플로를 피한다.
- **파라메트릭 서치**는 "값의 탐색"을 "답의 탐색"으로 바꾸는 발상. 단조성이 있으면 성립.
- 뺄셈 비교(`a - b`)는 오버플로 위험 → `Integer.compare(a, b)`.
