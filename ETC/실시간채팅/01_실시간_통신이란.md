# 01. 실시간 통신이란 — HTTP만으로는 왜 부족한가

> **이 문서에서 배우는 것**
> - HTTP의 "요청 → 응답 → 끊김" 방식으로는 왜 실시간 채팅이 어려운지
> - 억지로 실시간처럼 흉내 내는 방법(폴링)과 그 한계
> - **WebSocket**이 무엇이고, HTTP와 어떻게 다른지
> - 연결이 맺어지는 과정(핸드셰이크)을 큰 그림으로

---

## 1. HTTP의 근본 성질 — "한 번 묻고, 한 번 답하면 끝"

지금까지 만든 모든 앱은 이렇게 동작했습니다.

```
[브라우저] ──── 요청 ────▶ [서버]
          ◀─── 응답 ─────
          (여기서 연결 종료)
```

이 방식의 특징은 두 가지입니다.

1. **브라우저가 먼저 물어야만** 서버가 답할 수 있습니다. 서버가 먼저 말을 걸 방법이 없습니다.
2. 답을 주고 나면 **연결은 곧바로 끊깁니다.** 서버는 방금 누가 다녀갔는지도 기억하지 않습니다.

채팅을 생각해 봅시다. 영희가 메시지를 보냈을 때, **철수는 아무 요청도 하지 않았습니다.**
그런데 철수 화면에 영희의 말이 떠야 합니다. HTTP의 규칙("브라우저가 물어야 답한다")과 정면으로 부딪힙니다.

> 📌 이것이 핵심 문제입니다. 실시간이란 **"서버가 브라우저에게 먼저 말을 걸 수 있어야 한다"** 는 뜻인데,
> 순수 HTTP는 그 능력이 없습니다.

---

## 2. 억지로 흉내 내기 — 폴링(Polling)

그렇다면 브라우저가 **"새 메시지 있어요?"** 라고 계속 물어보면 되지 않을까요? 맞습니다. 이것이 **폴링**입니다.

```
브라우저: 새 거 있어? → 서버: 없음
(1초 뒤)
브라우저: 새 거 있어? → 서버: 없음
(1초 뒤)
브라우저: 새 거 있어? → 서버: "영희: 안녕" 있음!
```

동작은 합니다. 하지만 문제를 냉정하게 따져 보면 실무에서 쓰기 어렵습니다.

| 방식 | 실시간성 | 서버 부담 | 낭비 | 구현 난이도 |
|---|---|---|---|---|
| **짧은 폴링**(1초마다 질문) | 최대 1초 지연 | **높음** (질문이 폭주) | 대부분 "없음" 응답 = 대부분 헛질문 | 쉬움 |
| **롱 폴링**(답이 생길 때까지 붙잡기) | 빠름 | 중간 | 연결을 계속 다시 맺음 | 중간 |
| **WebSocket** | 즉시 | **낮음** | 필요할 때만 오감 | 중간 |

> ⚠️ **폴링의 진짜 비용은 "곱하기"에서 옵니다.** 접속자가 1,000명이고 1초마다 질문하면,
> 실제 대화가 없어도 서버는 **초당 1,000번의 헛요청**을 받아 냅니다. 대부분이 "새 거 없음" 응답이죠.
> 사용자가 적은 사내 도구라면 폴링도 "충분히 쓸 만한 실용적 선택"일 수 있지만,
> 사람이 많아질수록 급격히 불리해집니다.

폴링은 "HTTP의 한계를 노력으로 때우는" 방법입니다. 근본적으로 다른 통신 방식이 있으면 좋겠죠. 그것이 WebSocket입니다.

---

## 3. WebSocket — 통로를 열어 두고 양쪽이 자유롭게

WebSocket은 한 번 연결하면 그 **통로(연결)를 끊지 않고 계속 열어 둡니다.**
그 통로 위에서는 **브라우저도, 서버도, 아무 때나 먼저 말을 걸 수 있습니다.** (양방향)

```
[브라우저] ══════ 항상 열려 있는 통로 ══════ [서버]
    │  "안녕" 보냄 ───────────────────▶       │
    │       ◀─────────────── "영희: 안녕" 보냄 │   ← 서버가 먼저!
    │       ◀─────────────── "철수 입장" 보냄  │   ← 요청 안 했는데 옴
```

전화 통화를 떠올리면 쉽습니다.

- **HTTP** = 문자 메시지. 한 통 보내고 답이 오면 끝. 매번 새로 건다.
- **WebSocket** = 전화 통화. 한 번 연결하면 끊기 전까지 양쪽이 자유롭게 말한다.

이 "열린 통로" 덕분에 서버는 영희의 메시지가 도착하는 즉시, 접속 중인 철수에게 **먼저 밀어 넣을 수** 있습니다.
철수는 아무것도 안 했는데 화면에 뜨는 것이죠. 이것이 실시간의 정체입니다.

---

## 4. 연결은 어떻게 시작되나 — 핸드셰이크(Handshake)

재미있는 점은, WebSocket 연결도 **첫 시작은 평범한 HTTP 요청**이라는 것입니다.

```
1) 브라우저:  "나 이 연결을 WebSocket으로 바꾸고 싶어요"
              (일반 HTTP 요청에 Upgrade: websocket 이라는 표시를 담아 보냄)
                         │
                         ▼
2) 서버:      "좋아요, 바꿉시다" (101 Switching Protocols 응답)
                         │
                         ▼
3) 이 순간부터 그 연결은 HTTP가 아니라 WebSocket 통로가 됩니다. (그리고 계속 열려 있음)
```

이 "업그레이드" 과정을 **핸드셰이크**라고 부릅니다. 악수 한 번 하고 나면 관계(연결)가 바뀌는 것이죠.
그래서 주소도 `http://` 가 아니라 `ws://` 를 씁니다. (HTTPS 환경이면 `wss://` — 보안 연결)

```
http://localhost:8080   →   ws://localhost:8080/ws/chat
https://myapp.com        →   wss://myapp.com/ws/chat
```

> 📌 핸드셰이크가 HTTP로 시작하기 때문에, 기존 웹 서버·인증·방화벽 설정을 상당 부분 **재활용**할 수 있습니다.
> 완전히 새로운 별개의 기술이 아니라, HTTP 위에서 자연스럽게 "승격"되는 구조라고 이해하면 됩니다.

---

## 5. 언제 WebSocket을 쓰고, 언제 쓰지 말아야 하나

WebSocket이 항상 정답은 아닙니다. **"서버가 먼저, 자주, 즉시 말을 걸어야 하는가?"** 로 판단하세요.

| 상황 | 적합한 방식 | 이유 |
|---|---|---|
| 채팅, 실시간 알림, 협업 편집, 게임 | **WebSocket** | 서버가 수시로 먼저 밀어 줘야 함 |
| 주가·센서값을 1초마다 갱신 | WebSocket 또는 SSE | 지속적인 서버→클라 흐름 |
| 게시글 목록, 상세 조회, 저장 | **그냥 HTTP** | 사용자가 요청할 때만 필요 |
| 몇 분에 한 번 상태만 확인 | **폴링으로 충분** | 굳이 통로를 유지할 필요 없음 |

> ⚠️ **"실시간처럼 보인다"고 다 WebSocket이 필요한 건 아닙니다.** 갱신 주기가 길거나(수 분),
> 서버가 먼저 말 걸 일이 없다면 HTTP/폴링이 더 단순하고 안전합니다. 열린 통로는 **유지 비용**이 있으니까요
> (서버는 접속자 수만큼 연결을 계속 붙들고 있어야 합니다 — 03 문서의 "확장" 참고).

> 📌 **참고: SSE(Server-Sent Events)** 라는 중간 선택지도 있습니다. "서버 → 브라우저" **한 방향**만
> 필요할 때(알림·피드) 쓰는, WebSocket보다 가벼운 방식입니다. 채팅처럼 **양방향**이 필요하면
> WebSocket이 맞습니다. (이 모듈은 양방향이므로 WebSocket을 씁니다.)

---

## 정리

- HTTP는 "브라우저가 물어야 답하고, 답하면 끊긴다" — 서버가 먼저 말을 걸 수 없어 실시간에 불리합니다.
- **폴링**은 계속 물어보는 우회법. 간단하지만 사람이 많아지면 헛요청이 폭증합니다.
- **WebSocket**은 통로를 열어 두고 양쪽이 자유롭게 주고받는 방식. 서버가 **먼저** 밀어 줄 수 있어 채팅에 딱 맞습니다.
- 연결은 HTTP 요청으로 시작해 **핸드셰이크**로 승격되며, 주소는 `ws://` / `wss://` 를 씁니다.
- 도구를 고를 땐 "서버가 먼저·자주·즉시 말을 거는가?"를 기준으로. 아니라면 HTTP가 더 단순한 정답입니다.
