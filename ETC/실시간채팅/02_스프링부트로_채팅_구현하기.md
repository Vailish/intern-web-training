# 02. 스프링부트로 채팅 구현하기 — 코드 한 줄씩

> **이 문서에서 배우는 것**
> - WebSocket에 필요한 의존성과, 주소를 핸들러에 연결하는 **설정**
> - 연결·메시지·종료 3가지 사건을 처리하는 **핸들러(채팅의 심장)**
> - 브라우저와 서버가 주고받는 메시지의 모양(**JSON ↔ 자바 객체**)
> - 브라우저 내장 `WebSocket`으로 만든 **프론트엔드**
> - 입장 → 대화 → 퇴장이 흐르는 전체 그림

> 결론부터: 서버 쪽에서 **새로 배우는 것은 딱 2개**입니다 — `WebSocketConfig`(주소 등록)와
> `ChatWebSocketHandler`(사건 처리). 나머지(DTO, 프론트엔드)는 지금까지 쓰던 개념 그대로입니다.

이 문서는 `샘플/chat` 의 실제 코드를 그대로 따라갑니다. 파일을 열어 두고 함께 읽으면 좋습니다.

---

## 0. 전체 흐름 먼저 (숲을 보고 나무로)

```
① 브라우저가 ws://.../ws/chat 로 연결          → 서버: afterConnectionEstablished (연결됨)
② 브라우저가 {type:ENTER, sender:"철수"} 전송  → 서버: handleTextMessage → "철수 입장" 을 전원에 방송
③ 브라우저가 {type:TALK, content:"안녕"} 전송   → 서버: handleTextMessage → "철수: 안녕" 을 전원에 방송
④ 브라우저 탭을 닫음                           → 서버: afterConnectionClosed → "철수 퇴장" 을 전원에 방송
```

`②③④` 에서 서버가 하는 일은 똑같습니다: **받은 메시지를 접속자 전원에게 복사해서 보낸다(브로드캐스트).**
이 한 문장이 그룹 채팅의 전부입니다. 이제 이걸 코드로 확인합니다.

## 2. 주소를 핸들러에 연결 — `config/WebSocketConfig.java`

HTTP에서 `@GetMapping("/intro")` 로 주소와 처리기를 연결했듯이, WebSocket에서는 이 설정 파일이 그 역할을 합니다.

```java
@Configuration
@EnableWebSocket // 이 앱에서 WebSocket 기능을 켭니다.
public class WebSocketConfig implements WebSocketConfigurer {

    private final ChatWebSocketHandler chatWebSocketHandler;

    public WebSocketConfig(ChatWebSocketHandler chatWebSocketHandler) {
        this.chatWebSocketHandler = chatWebSocketHandler; // 지금까지 서비스 주입과 동일
    }

    @Override
    public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
        registry.addHandler(chatWebSocketHandler, "/ws/chat");
        // "/ws/chat 으로 들어온 WebSocket 연결은 chatWebSocketHandler 가 맡는다"
    }
}
```

읽는 법은 단 한 줄입니다: **"`/ws/chat` 주소로 연결되면 `ChatWebSocketHandler` 가 처리한다."**
이 주소는 프론트엔드의 `new WebSocket(".../ws/chat")` 과 **글자 하나까지 똑같아야** 합니다.

> 🏢 **eGovFrame 연결 포인트** — 레거시 Spring MVC(XML 설정)에서는 이 등록을
> `<websocket:handlers>` 태그로 합니다. 개념(주소 ↔ 핸들러 매핑)은 동일하고, 표현 방식만 다릅니다.

> 📌 **허용 출처(Origin) 이야기** — 이 예제는 서버가 채팅 화면(`index.html`)까지 함께 내려 주므로
> 화면과 서버가 **같은 출처**(`http://localhost:8080`)입니다. 그래서 별도 허용 설정이 필요 없습니다.
> 만약 React(예: `localhost:5173`)처럼 **다른 곳**에서 접속한다면 `.setAllowedOrigins("http://localhost:5173")`
> 로 그 주소를 명시해야 합니다. 이때 귀찮다고 `"*"`(전부 허용)로 열면 안 됩니다 —
> [Front-Back 모듈](../../Spring/Front-Back/README.md)의 CORS "와일드카드 금지"와 정확히 같은 이유입니다.

## 3. 오가는 메시지의 모양 — `dto/ChatMessage.java`

브라우저(자바스크립트)와 서버(자바)는 언어가 다릅니다. 둘의 공통 약속이 **JSON**입니다.

```
브라우저가 보내는 JSON:  {"type":"TALK", "sender":"철수", "content":"안녕하세요"}
                              │  스프링이 자동 변환(Jackson)
                              ▼
서버 안의 자바 객체:       ChatMessage(type=TALK, sender="철수", content="안녕하세요")
```

그래서 이 모양을 담을 상자(DTO)를 하나 만듭니다.

```java
public class ChatMessage {
    public enum Type { ENTER, TALK, LEAVE } // 입장 / 대화 / 퇴장

    private Type type;       // 메시지 종류
    private String sender;   // 보낸 사람 닉네임
    private String content;  // 메시지 내용
    private int userCount;   // 지금 접속 인원 (화면 상단 "N명 접속중"용)

    public ChatMessage() {}  // Jackson이 객체를 만들 때 기본 생성자가 필요
    // 이하 getter / setter (생략)
}
```

`type` 으로 "이게 입장 알림인지, 대화인지, 퇴장 알림인지" 를 구분하는 것이 설계의 핵심입니다.
같은 통로로 3가지 종류의 메시지가 흐르기 때문에, 받는 쪽이 종류를 보고 다르게 처리할 수 있어야 합니다.

> 📌 필드를 `private` 로 감추고 `getter`/`setter` 로 여닫는 이 모양은 회사 실무(eGovFrame)의
> **VO/DTO** 와 똑같습니다. 새로운 문법이 아니라, 익숙한 자바빈(JavaBean) 그대로입니다.

## 4. 채팅의 심장 — `handler/ChatWebSocketHandler.java`

`TextWebSocketHandler` 를 상속하면, **"사건이 일어났을 때 불릴 메서드"** 만 채우면 됩니다. (콜백 방식)

### 4-1. 접속자 목록 들고 있기

```java
@Component
public class ChatWebSocketHandler extends TextWebSocketHandler {

    // 지금 접속한 모든 연결(세션). 방송할 때 이 목록을 훑는다.
    private final Set<WebSocketSession> sessions = new CopyOnWriteArraySet<>();

    // 세션 → 닉네임. 연결이 끊길 때 "누가 나갔는지" 이름을 찾기 위해.
    private final Map<String, String> nicknames = new ConcurrentHashMap<>();
```

> ⚠️ **왜 그냥 `HashSet` 이 아니라 `CopyOnWriteArraySet` 인가?** 채팅 서버에는 여러 사람이
> **동시에** 접속하고 나갑니다. 즉 여러 스레드가 이 목록을 **한꺼번에** 건드립니다. 평범한 `HashSet` 을
> 여러 스레드가 동시에 수정하면 데이터가 깨지거나 예외가 납니다. `CopyOnWriteArraySet` /
> `ConcurrentHashMap` 은 이런 동시 접근에 안전하도록 만들어진 자료구조입니다. (실무에서 놓치기 쉬운 부분)

### 4-2. [사건 1] 연결됨

```java
@Override
public void afterConnectionEstablished(WebSocketSession session) {
    sessions.add(session);
    // 아직 닉네임은 모른다. 곧 클라이언트가 ENTER 메시지로 알려 준다.
}
```

전화가 연결된 순간입니다. 일단 목록에 등록만 해 둡니다.

### 4-3. [사건 2] 메시지 도착 (알맹이)

```java
@Override
protected void handleTextMessage(WebSocketSession session, TextMessage message) throws Exception {
    // 도착한 JSON 문자열 → 자바 객체
    ChatMessage chatMessage = objectMapper.readValue(message.getPayload(), ChatMessage.class);

    if (chatMessage.getType() == ChatMessage.Type.ENTER) {
        // 입장: 닉네임을 기억하고, 안내 문구를 서버가 만들어 준다.
        nicknames.put(session.getId(), chatMessage.getSender());
        chatMessage.setContent(chatMessage.getSender() + "님이 입장했습니다.");
    }
    // TALK(대화)이면 content 를 그대로 둔다.

    chatMessage.setUserCount(sessions.size()); // 현재 인원 실어서
    broadcast(chatMessage);                    // 전원에게 방송
}
```

입장 메시지일 때만 서버가 안내 문구("○○님이 입장했습니다")를 대신 만들어 줍니다.
일반 대화는 내용을 건드리지 않고 그대로 방송합니다.

### 4-4. [사건 3] 연결 끊김

```java
@Override
public void afterConnectionClosed(WebSocketSession session, CloseStatus status) throws Exception {
    sessions.remove(session);
    String nickname = nicknames.remove(session.getId());
    if (nickname != null) { // 정상 입장했던 사람이면 퇴장 알림
        ChatMessage leave = new ChatMessage();
        leave.setType(ChatMessage.Type.LEAVE);
        leave.setSender(nickname);
        leave.setContent(nickname + "님이 나갔습니다.");
        leave.setUserCount(sessions.size());
        broadcast(leave);
    }
}
```

탭을 닫거나 새로고침하면 통로가 끊기고 이 메서드가 불립니다. 목록에서 지우고 남은 이들에게 알립니다.
연결이 끊기는 시점엔 메시지 내용이 없으므로, **아까 저장해 둔 닉네임**을 여기서 꺼내 씁니다.

### 4-5. 전원에게 방송 — `broadcast`

```java
private void broadcast(ChatMessage chatMessage) throws Exception {
    String json = objectMapper.writeValueAsString(chatMessage); // 자바 객체 → JSON 문자열
    TextMessage textMessage = new TextMessage(json);

    for (WebSocketSession session : sessions) {   // 접속자 전원에게
        if (session.isOpen()) {
            synchronized (session) {               // 같은 세션에 동시 전송 방지
                session.sendMessage(textMessage);
            }
        }
    }
}
```

**목록을 훑으며 똑같은 메시지를 모두에게 보낸다** — 이것이 그룹 채팅의 정체입니다.

> ⚠️ **`synchronized (session)` 은 왜?** 하나의 세션에 두 스레드가 **동시에** 메시지를 쓰면
> `TEXT_PARTIAL_WRITING` 같은 오류가 날 수 있습니다. 세션 단위로 잠금을 걸어 "한 세션에는 한 번에 하나씩"
> 보내도록 막았습니다. 간단한 예제라도 이런 견고성(robustness)을 챙기는 습관이 중요합니다.

## 5. ⚠️ 스프링부트 4의 JSON 함정 — Jackson 3

JSON ↔ 자바 변환기를 만드는 코드는 원래 이렇게 씁니다(인터넷 예제 대부분).

```java
import com.fasterxml.jackson.databind.ObjectMapper;   // ← 스프링부트 3.x 까지
private final ObjectMapper objectMapper = new ObjectMapper();
```

그런데 **스프링부트 4.x** 에서는 그대로 붙여넣으면 **컴파일 오류**(`package ... does not exist`)가 납니다.
부트 4부터 JSON 라이브러리 **Jackson이 3버전**으로 올라가면서 패키지 이름이 통째로 바뀌었기 때문입니다.

```java
import tools.jackson.databind.ObjectMapper;            // ← 스프링부트 4.x (com.fasterxml → tools.jackson)
import tools.jackson.databind.json.JsonMapper;
private final ObjectMapper objectMapper = JsonMapper.builder().build();  // new ObjectMapper() 대신 권장
```

| | 스프링부트 3.x (Jackson 2) | 스프링부트 4.x (Jackson 3) |
|---|---|---|
| 패키지 | `com.fasterxml.jackson.databind` | `tools.jackson.databind` |
| 객체 생성 | `new ObjectMapper()` | `JsonMapper.builder().build()` |

> ⚠️ **인터넷 예제를 그대로 믿지 마세요.** 이 프로젝트는 부트 4.1로 실행 검증했고, 위 함정 때문에
> 실제로 처음 컴파일이 실패했습니다. 검색해서 나오는 코드 대부분은 아직 부트 3.x 기준(`com.fasterxml...`)이라
> **버전이 다르면 그대로는 안 됩니다.** "왜 import가 빨간 줄이지?" 싶을 때 이 표를 떠올리세요.
> (신뢰수준 고지: 이 표는 검증된 사실이지만, 라이브러리는 계속 바뀝니다. 정확한 것은 항상
> 내 프로젝트의 `gradlew dependencies` 로 실제 버전을 확인하는 것입니다.)

## 6. 프론트엔드 — `static/index.html` (브라우저 내장 WebSocket)

프론트엔드에는 프레임워크가 없습니다. 브라우저에 원래 들어 있는 `WebSocket` 만 씁니다.
`fetch` 를 안다면 절반은 온 것입니다 — **"fetch는 한 번 묻고 끝, WebSocket은 통로를 계속 열어 둔다"**.

### 6-1. 연결 열기

```javascript
// http → ws, https → wss 로 바뀌는 것이 규칙
const protocol = location.protocol === 'https:' ? 'wss' : 'ws';
socket = new WebSocket(protocol + '://' + location.host + '/ws/chat');
//                                                        └─ 서버 WebSocketConfig의 주소와 동일해야 함
```

### 6-2. 4가지 사건에 반응하기

```javascript
socket.onopen  = () => sendToServer({ type: 'ENTER', sender: myName, content: '' });
socket.onmessage = (event) => {            // ★ 핵심: 내가 요청 안 해도 서버가 보내면 실행됨
    const message = JSON.parse(event.data); // 받은 JSON 문자열 → 객체
    renderMessage(message);                  // 화면에 그리기
    userCountLabel.textContent = message.userCount + '명 접속중';
};
socket.onclose = () => renderSystem('연결이 종료되었습니다.');
socket.onerror = () => renderSystem('연결 오류가 발생했습니다.');
```

`onmessage` 가 실시간의 핵심입니다. **버튼을 누르거나 요청을 보내지 않아도**, 서버가 통로로
메시지를 밀어 넣는 순간 이 함수가 저절로 실행됩니다. `fetch` 에는 없는, WebSocket만의 능력입니다.

### 6-3. 메시지 보내기

```javascript
function sendMessage() {
    sendToServer({ type: 'TALK', sender: myName, content: messageInput.value.trim() });
}
function sendToServer(obj) {
    socket.send(JSON.stringify(obj)); // 객체 → JSON 문자열로 바꿔 통로에 흘려보냄
}
```

### 6-4. 작지만 중요한 보안 습관 — XSS 방지

```javascript
function escapeHtml(text) {
    const div = document.createElement('div');
    div.textContent = text;   // 텍스트로 넣으면 <script> 같은 태그가 "글자"로 취급됨
    return div.innerHTML;      // → 안전하게 표시용 문자로 변환
}
```

> ⚠️ 사용자가 친 글을 화면에 그대로 HTML로 꽂으면, 누군가 `<script>...</script>` 를 채팅으로 보내
> 남의 브라우저에서 악성 코드를 실행시킬 수 있습니다(**XSS**). 채팅처럼 "남이 입력한 값을 남에게 보여 주는"
> 기능은 XSS의 대표적 표적이므로, 표시 전에 반드시 위처럼 무해화(escape)해야 합니다.

## 7. 직접 실행하고 확인하기

```powershell
cd 샘플\chat
.\gradlew.bat bootRun
```

1. `http://localhost:8080` 접속 → `철수` 로 입장.
2. **새 탭**을 하나 더 열어 `영희` 로 입장.
3. 한쪽에서 메시지를 보내면 **양쪽에 동시에** 뜨는지 확인. (내 말은 오른쪽 파란색, 상대 말은 왼쪽 흰색)
4. 한쪽 탭을 닫으면 다른 쪽에 **"○○님이 나갔습니다"** + 접속 인원이 줄어드는지 확인.

이 4단계가 모두 되면, 앞에서 본 `연결 → 입장 → 대화 → 퇴장` 흐름을 두 눈으로 확인한 것입니다.

## 정리

- WebSocket 서버에서 새로 배우는 건 **설정(주소 등록)** 과 **핸들러(사건 처리)** 2개뿐입니다.
- 핸들러의 3대 사건: **연결됨 / 메시지 도착 / 연결 끊김**. 여기에 방송(broadcast)을 더하면 채팅 완성.
- 브라우저·서버는 **JSON** 으로 대화하며, 스프링이 자바 객체(DTO)와 자동 변환해 줍니다.
- 부트 4의 **Jackson 3 패키지 변경**(`com.fasterxml` → `tools.jackson`)은 꼭 기억할 함정입니다.
- 프론트엔드는 내장 `WebSocket` 의 `onopen/onmessage/onclose/onerror` 4개로 충분하며,
  `onmessage` 가 "서버가 먼저 밀어 주는 실시간"의 핵심입니다.
- 남의 입력을 보여 주는 기능에는 **XSS 방지**가 기본기입니다.
