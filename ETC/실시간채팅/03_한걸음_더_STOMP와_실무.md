# 03. 한 걸음 더 — STOMP와 실무 고려사항

> **이 문서에서 배우는 것**
> - 지금 만든 채팅이 "실무에서 부족한 점"을 냉정하게 짚기
> - 채팅방 여러 개, 메시지 저장, 인증을 붙이려면 어디를 손대야 하는지
> - 서버를 여러 대로 늘리면 생기는 **치명적 문제**와 해결책(메시지 브로커)
> - 실무에서 널리 쓰는 **STOMP** 방식이란 무엇이고, 언제 갈아탈지

> 결론부터: 이 모듈에서 만든 원시(raw) WebSocket 채팅은 **원리를 이해하기엔 최고**지만,
> 실서비스에는 4가지가 빠져 있습니다 — **① 방 구분 ② 저장 ③ 인증 ④ 서버 확장.**
> 이 중 **④ 확장**이 가장 위험하며, 여기서 STOMP + 메시지 브로커가 등장합니다.

---

## 1. 지금 만든 채팅의 한계 — 정직하게 짚기

잘 동작하지만, 실무에 그대로 올리면 곤란한 지점들이 있습니다. 우선순위(위험한 순서)로 정리합니다.

| 순위 | 한계 | 지금 상태 | 실무에서 필요한 것 |
|---|---|---|---|
| 1 | **서버 1대 전제** | 접속자를 한 서버의 메모리(`Set`)에 저장 | 서버 여러 대에서도 메시지가 오가야 함 (§5) |
| 2 | **인증 없음** | 닉네임을 스스로 신고(아무나 "관리자" 사칭 가능) | 로그인한 사용자만, 신원 검증 (§4) |
| 3 | **저장 안 됨** | 새로고침하면 이전 대화가 사라짐 | 메시지를 DB에 남기기 (§3) |
| 4 | **방이 하나** | 모두가 한 방에서 대화 | 방(채팅룸) 여러 개 (§2) |

> 📌 **이것은 설계 실패가 아니라 의도된 단순화입니다.** 원리를 가리는 곁가지를 걷어낸 것이죠.
> 아래에서 하나씩 "어디를 어떻게 손대면 되는지" 방향만 잡아 줍니다. (전체 구현은 각자의 다음 과제)

---

## 2. 채팅방 여러 개 만들기

지금은 접속자를 하나의 `Set` 에 담아 전원에게 방송했습니다. 방을 나누려면 **"방 이름 → 그 방의 접속자 목록"** 구조로 바꾸면 됩니다.

```java
// 지금:  모두 한 곳에
private final Set<WebSocketSession> sessions = new CopyOnWriteArraySet<>();

// 방 구분:  방 이름별로 접속자 목록을 따로
private final Map<String, Set<WebSocketSession>> rooms = new ConcurrentHashMap<>();
```

- 연결 주소에 방을 실어 보냅니다: `ws://.../ws/chat?roomId=general`
- 방송할 때 **전원**이 아니라 **같은 방 사람들에게만** 보냅니다.

원리는 그대로고, "누구에게 보낼지" 범위만 좁아졌을 뿐입니다.

---

## 3. 메시지 저장하기 (대화 기록 남기기)

지금은 메시지를 흘려보내기만 합니다. 기록을 남기려면 [JPA 교육](../../Spring/Database/README.md)에서 배운 그대로,
방송 **직전에** 한 줄 저장하면 됩니다.

```java
// handleTextMessage 안에서, broadcast 하기 전에:
chatMessageRepository.save(new ChatLog(sender, content, /* 시각 */));
broadcast(chatMessage);
```

이러면 build.gradle에 다시 **JPA·DB 의존성이 등장**합니다. (이 모듈에서 뺐던 것들이 실무에선 대개 돌아옵니다.)
입장 시 최근 N개를 조회해 화면에 먼저 뿌려 주면 "이전 대화 이어 보기"도 됩니다.

---

## 4. 인증 붙이기 — "닉네임 신고제"의 위험

> ⚠️ **지금 구조의 가장 큰 보안 구멍:** 클라이언트가 `sender:"관리자"` 라고 **스스로 주장**하면
> 서버가 그대로 믿습니다. 아무나 남을 사칭할 수 있다는 뜻입니다. 실서비스에선 절대 안 됩니다.

해결의 핵심은 **"닉네임을 클라이언트가 보내게 하지 말고, 서버가 로그인 정보에서 꺼내 쓰는 것"** 입니다.

- 핸드셰이크(§01의 4절)는 HTTP로 시작하므로, 이때 **세션/토큰으로 로그인 여부를 검사**할 수 있습니다.
- `HandshakeInterceptor` 로 "로그인 안 했으면 연결 거부", 통과하면 검증된 사용자 정보를 세션에 심어 둡니다.
- 이후 메시지의 `sender` 는 클라이언트 말이 아니라 **서버가 심어 둔 신원**을 씁니다.

> 🏢 **eGovFrame 연결 포인트** — 회사 표준의 로그인/권한 체계(Spring Security 기반)와 WebSocket을
> 연동할 때 바로 이 지점을 씁니다. "누가 접속했는가"를 프레임워크의 인증 정보에서 가져오는 것이죠.

## 5. 서버를 여러 대로 — 가장 위험하고, 가장 중요한 문제

접속자가 늘면 서버를 여러 대로 늘립니다(부하 분산). 그런데 여기서 **원시 WebSocket 채팅이 조용히 깨집니다.**

```
        [서버 A]  ← 철수, 영희 접속        [서버 B]  ← 민수 접속
             │                                 │
철수가 "안녕" → 서버 A의 Set(철수,영희)에게만 방송 → 민수(서버 B)는 못 받음! ❌
```

철수의 메시지는 **철수가 붙어 있는 서버 A의 메모리 목록**에만 뿌려집니다.
민수는 서버 B에 붙어 있으니 그 목록에 없어서 **영영 메시지를 못 받습니다.** 접속자 목록이 서버마다 따로 놀기 때문입니다.

**해결책: 서버들 바깥에 공용 "메시지 중계소(브로커)"를 둡니다.** (Redis Pub/Sub, RabbitMQ, Kafka 등)

```
[서버 A] ─┐                          ┌─▶ [서버 A] → 철수·영희에게
          ├─▶ [메시지 브로커(Redis)] ─┤
[서버 B] ─┘   (모든 서버가 여기 구독)   └─▶ [서버 B] → 민수에게

철수 "안녕" → 서버 A → 브로커 → 모든 서버가 전달받음 → 각 서버가 자기 접속자에게 방송 → 전원 수신 ✔
```

> 📌 **핵심 통찰:** 단일 서버에서는 우리가 만든 `Set` 이 곧 브로커였습니다. 서버가 여러 대가 되는 순간,
> 그 역할을 **서버 바깥의 공용 브로커**가 대신해야 합니다. 이 필요성이 다음 절의 STOMP로 자연스럽게 이어집니다.

## 6. STOMP — 실무에서 널리 쓰는 방식

우리는 "메시지 종류(ENTER/TALK/LEAVE)", "방 구분", "누구에게 보낼지"를 **직접 코드로** 관리했습니다.
규모가 커지면 이런 규칙을 매번 손으로 짜는 게 부담됩니다. **STOMP** 는 이 규칙을 표준화한 약속(프로토콜)입니다.

STOMP의 모델은 **구독(subscribe) / 발행(publish)** 입니다. 유튜브 채널을 떠올리면 쉽습니다.

```
구독:  클라이언트가 "/topic/room1 채널을 구독할게요"   → 그 채널에 뜨는 건 자동으로 받음
발행:  누군가 "/topic/room1 에 메시지 올림"           → 구독한 전원에게 자동 전달
```

스프링부트 설정도 간결해집니다(개념 예시).

```java
@Configuration
@EnableWebSocketMessageBroker
public class StompConfig implements WebSocketMessageBrokerConfigurer {
    @Override public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws-stomp"); // 연결 지점
    }
    @Override public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/topic");     // 구독 주소 접두어
        registry.setApplicationDestinationPrefixes("/app"); // 보낼 주소 접두어
    }
}

@Controller
class ChatController {
    @MessageMapping("/chat/{roomId}")   // 클라이언트가 여기로 보내면
    @SendTo("/topic/{roomId}")          // 이 채널 구독자 전원에게 자동 전달 (broadcast 반복문 불필요!)
    public ChatMessage handle(ChatMessage msg) { return msg; }
}
```

우리가 손으로 짠 `sessions` 순회·`broadcast` 반복문이 `@SendTo` 한 줄로 사라진 것이 눈에 띕니다.
게다가 위 5절의 **여러 대 서버 문제**도, `enableSimpleBroker` 를 외부 브로커(RabbitMQ 등)로 바꾸면
설정만으로 해결됩니다. STOMP가 실무에서 사랑받는 이유입니다.

### 원시 WebSocket vs STOMP — 언제 무엇을?

| 지표 | 원시 WebSocket (이 모듈) | STOMP + 브로커 |
|---|---|---|
| **학습·이해** | 쉬움 — 동작이 다 보임 | 추상화가 있어 처음엔 낯섦 |
| **초기 구현량** | 적음 (핸들러 1개) | 설정·의존성이 더 필요 |
| **방/구독 관리** | 직접 코드로 | 프로토콜이 기본 제공 |
| **서버 여러 대 확장** | **직접 브로커 연동 필요(어려움)** | **설정으로 해결(쉬움)** |
| **적합한 상황** | 학습, 단순·소규모, 완전한 제어가 필요할 때 | 방 많은 대규모, 팀 협업 실서비스 |

> 📌 **순서를 지켜 배우는 것이 이득입니다.** STOMP를 먼저 배우면 "마법처럼 되긴 하는데 안이 안 보이는"
> 상태가 됩니다. 이 모듈처럼 원시 WebSocket으로 **바닥(연결·세션·방송)** 을 손에 익힌 뒤 STOMP로 올라가면,
> `@SendTo` 한 줄 뒤에서 무슨 일이 벌어지는지 훤히 보입니다.

## 7. 그 밖의 실무 체크리스트

- **연결 끊김 자동 복구:** 지하철에서 네트워크가 끊기면 통로도 끊깁니다. 프론트엔드에서 `onclose` 시
  잠시 후 **자동 재연결**을 시도하는 코드가 실무엔 거의 항상 들어갑니다.
- **하트비트(heartbeat):** "살아 있나요?" 를 주기적으로 주고받아, 죽은 연결을 빨리 감지하고 정리합니다.
- **메시지 크기·빈도 제한:** 거대한 메시지나 초당 수백 건의 도배를 막아 서버를 보호합니다(입력 검증·rate limit).
- **보안(재확인):** `wss://`(암호화) 사용, 허용 출처 명시([02 문서](./02_스프링부트로_채팅_구현하기.md)),
  화면 출력 시 XSS 방지 — 실시간 기능은 "남의 입력을 남에게 즉시 보여 주는" 특성상 특히 신경 써야 합니다.

## 정리

- 지금 만든 채팅의 실무 공백 4가지: **① 서버 확장 ② 인증 ③ 저장 ④ 방 구분** — 위험한 순서로 우선순위를 두세요.
- 방·저장·인증은 지금 구조에 **부분적으로 덧붙이면** 됩니다(원리는 그대로).
- 서버 여러 대는 접속자 목록이 서버마다 갈라져 **메시지가 새는** 문제가 있고, **외부 메시지 브로커**로 풉니다.
- **STOMP** 는 구독/발행 모델로 방송·방·확장을 표준화한 방식. 대규모 실서비스에 유리합니다.
- 다만 **원시 WebSocket으로 바닥을 먼저 익힌 지금의 순서**가, STOMP를 제대로 이해하는 가장 빠른 길입니다.

---

## 이 모듈을 마치며

- HTTP와 WebSocket의 차이, 그리고 실시간이 왜 어려운지를 설명할 수 있습니다.
- 스프링부트에서 **연결·메시지·종료 + 방송**으로 동작하는 채팅을 직접 만들어 봤습니다.
- 실무로 키울 때 마주칠 **확장·인증·저장·보안** 문제와, STOMP라는 다음 단계를 압니다.

이제 인턴 스스로의 다음 과제로, **"채팅방 여러 개"** 또는 **"대화 내용 DB 저장"** 중 하나를
이 샘플에 직접 붙여 보길 권합니다. 원리를 알았으니, 손대야 할 곳은 이미 위에 표시되어 있습니다.
